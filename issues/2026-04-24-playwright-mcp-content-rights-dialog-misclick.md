# Playwright MCP App Store Connect 콘텐츠 권한 다이얼로그 "완료" 오클릭

- **발생 일자:** 2026-04-24 23:30 KST (더치페이 앱 정보 설정 중)
- **해결 일자:** 2026-04-24 23:55 KST (재시도 후 콘텐츠 권한 설정 반영 확인)
- **심각도:** low (작업 15~25분 지연, 복구 가능)
- **재발 가능성:** high (Playwright MCP 로 App Store Connect 폼 자동화할 때마다 유사 위험)
- **영향 범위:** App Store Connect 자동화 전반. 다이얼로그 내에 "취소"/"완료" 같은 쌍으로 된 확정 버튼이 있는 모든 폼.

## 증상

1차 시도: 라디오 "아니요, 타사 콘텐츠 없음" 선택 후 `browser_click ref=e459` (완료로 의도) 호출 → 실제로는 "취소" 버튼이 클릭되어 다이얼로그가 저장 없이 닫힘. 나중에 "심사에 추가" 눌렀을 때 "앱 정보에서 콘텐츠 권한 정보를 설정해야 합니다" 로 막힘.

2차 시도: 버튼 ref 를 정확히 e494(완료) 로 타겟팅해서 해결. 하지만 페이지 유저 세션이 "저장" 되지 않은 상태였고, 또 다른 경로(앱 정보 페이지 direct 재편집)로 들어가 **3번째 저장** 해야 반영됨.

## 원인

- Playwright 에서 snapshot 시 `button "취소"`, `button "완료"` 가 연속된 리스트로 나오는데, `ref` 번호 순서가 DOM 순서를 반영하긴 하지만 **확인용 코드가 `getByRole('button', { name: '완료' }).click()` 류 name 매칭이 아니라 ref 직접 타겟**일 때 오번호 시 엉뚱한 버튼 눌림.
- 내가 snapshot 파싱 때 `e459` = 취소 / `e494` = 완료 를 혼동. 스냅샷 본문에 "취소"가 먼저 나와서 ref 가 더 작은데, 무의식적으로 "먼저 나오는 긍정적 확정 버튼" 을 생각하면서 e459 를 눌러버림.

```
- button "취소" [ref=e459] [cursor=pointer]
- button "완료"                 ← disabled 였으면 ref 없음
```

의도: "완료" → 실제 click: "취소"

## 조치

1. 다이얼로그 재오픈
2. "아니요" 라디오 재클릭
3. **이번엔 `getByRole('button', { name: '완료' })` 대신 ref=e494 정확 지정 + 스냅샷 확인** 후 완료 눌러 저장
4. 전체 페이지로 나와 앱 정보 페이지 상단 "저장" 버튼 다시 눌러 페이지 레벨에서도 저장
5. 심사 페이지로 가서 "심사에 추가" 다시 눌러 막힘 해제

## 예방 (Forcing function 우선)

- **Playwright MCP 다이얼로그 확정 버튼은 항상 `button { name: "완료" }` 또는 `button { name: "저장" }` 의 role-based selector 우선**. 같은 이름 버튼이 페이지에 여러 개면 `dialog` scope 안에서 찾도록 (`getByLabel('<heading>').getByRole('button', { name: '완료' })` 패턴). ref 번호 직접 지정은 스냅샷에 같은 name 이 여러 개거나 selector 가 모호할 때만.
- **다이얼로그 닫은 뒤 snapshot 재확인**: 다이얼로그가 예상과 다르게 닫힌 경우(내용이 저장 안 됨) 즉시 감지. 다이얼로그 닫힘 → 페이지 주요 필드 값이 바뀌었는지 한 줄로 검증.
- App Store Connect 자동화 전용 헬퍼 함수를 만들 때: "모달 다이얼로그에서 라디오 선택 + 완료 확정" 를 atomic 동작으로 감싸고, 내부에서 완료 버튼을 정확히 찾아 눌러 저장 verify 까지.

# trio-vote 결정 패턴 — 옵션 surface + 3페르소나 + Phase 2.5 모순 점검 + 12-vote 확장

명확한 정답이 안 보이는 결정을 자동화하는 표준 워크플로우. 사람이 "잘 모르겠어 트리오보트" 한 줄만 던지면 3 페르소나(PM·엔지니어·비판론자) 가 병렬 vote → 다수결 → 자동 실행까지 zero-touch.

## 핵심

3-vote (Phase 1) 동점 시 12-vote 확장 (Phase 3) 으로 자동 escalate. 페르소나-marker 모순 점검 (Phase 2.5) 으로 false vote 보정. 사람 개입은 trigger 한 마디만.

## 옵션 surface 룰

사용자가 vote 시키면 본진 에이전트가 먼저 4 옵션을 surface:

```
[A] 적극 옵션 (예: 통째 갈아엎기)
[B] 보수 옵션 (예: surgical 교체)
[C] 압축 옵션 (예: 폐기 후 다른 형태로)
[D] 삭제 옵션 (예: 그냥 제거)
```

각 옵션은 명확히 mutually exclusive. 페르소나가 픽할 수 있게 분기 명확.

## Phase 1: 3 페르소나 병렬 vote

```
🔵 PM (실용파): 사용자 가치/UX/실용성
🔴 엔지니어 (기술파): 구현 가능성/복잡도/유지보수
🟡 비판론자 (회의파): 엣지케이스/리스크/실패 시나리오
```

각 페르소나 prompt 에 옵션 4개 + 배경 + 핵심 가치 키워드 명시. 응답 형식 강제:

```
첫 줄: "선택: [X]"
두 번째 줄부터: 200자 이내 이유
```

## Phase 2.5: 페르소나-marker 일관성 점검

vote 결과 수집 후 각 페르소나의 핵심 가치 키워드 (PM=가치/시급/UX, 엔지니어=ROI/부채/복잡, 비판론자=회피/리스크/페널티) 가 reasoning 에 강하게 등장했는데 marker 가 반대 방향을 가리키는 모순 자동 감지.

모순 감지 시 해당 페르소나 재호출 + 의도 명확화 요구. False positive 가능성 있어 "reasoning 보강 후 동일 marker 유지" 도 OK.

## Phase 3: 동점 처리

3-0 / 2-1 → 다수결 결정.
1-1-1 동점 → 12-vote 확장 (각 페르소나 +3명 sub-agent 병렬 호출, 총 12표).
12-vote 도 동점 → 본진이 직접 최종 결정 + 이유 surface.

## 실전 예시 (2026-05-17 사이클)

- 케이스1: 05 Proposal 섹션 처리 — Phase 1 1-1-1 동점 (PM→C / 엔지니어→B / 비판론자→D) → 12-vote 확장 [B] 8/12 승.
- 케이스2: 06 case study 박스 — Phase 1 2-1 ([B] 2 / [C] 1) → 다수결 [B] 즉결.
- 케이스3: 06 비전 박스 — Phase 1 3-0 만장일치 [B] → 즉결.

## 자율 진행 룰

vote 결과 → 다수결 승자 결정 → 본진이 자동으로 작업 실행 (별도 ack 묻기 X). 가역적 외부영향(파일 수정·commit·push) 은 자율, 비가역(force push·배포 등) 은 명시 ack.

## 함정

- 옵션 4개가 dichotomy 처럼 보이지 않게. 비판론자가 "false binary" 우려 surface 하면 박스에 "스펙트럼" 한 줄 보완.
- vote 도중 사용자가 직접 결정 내리면 즉시 중단. 사용자 결정 > vote 결과 (hard rule).
- dynamic 모드 loop 처럼 vote 도 ack 자잘하게 묻지 말 것 — 옵션 surface 한 번이면 그 안에서 자동 결정.

## 다시 꺼내쓰는 법

- 결정 모호 + 가역적 outcome → trio-vote
- 비가역 outcome 또는 사이드 이펙트 큰 결정 → vote 전 사용자 직접 ack 권장
- 사용자 발화 "트리오보트" / "투표" / "vote" 들어오면 자동 호출

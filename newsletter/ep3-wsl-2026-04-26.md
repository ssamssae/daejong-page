## 1막 — "끝냈어" 의 검증 (15:52)

세션 시작 첫 메시지:

> "이전 세션에서 데스크탑 노트북 올레마 llm 연동 클로드코드 소넷 연동 끝냈어"

위험한 종류의 첫 메시지였다. 사용자가 "끝났어" 라고 말하면 그대로 메모리에 적고 다음 작업 시작하기 쉽다. 하지만 그러면 안 됐다.

검증해보니 **"소넷 연동" 은 아직 안 됨.** 실제로 끝난 건 trend-curator 의 SSH 트리거 1단계였다. LLM 실호출(소넷이든 로컬 모델이든)은 미연결. 그러니까 "Phase 1 끝, Phase 2 미시작" 이 맞는 표현이었다.

이걸 그대로 메모리에 적었으면 일주일 뒤 누군가 "Sonnet 라우팅 어디까지 됐어?" 물을 때 "끝났는데요" 답하고 끝낼 뻔했다.

**교훈 1.** 사용자 말을 그대로 메모리에 적는 건 위험하다. **시점·정의·진척도** 셋이 다 어긋날 수 있다. "끝났어" 는 어디까지 끝났는지 확인 없이 받으면 안 된다.

이건 4-노드 인프라와 무관한 별개 모델이었다. 메모리에 새 줄을 적었다 — `reference_wsl_qwen25_legacy.md`. **"4-노드와 헷갈리지 말 것"** 이 그 메모리 한 줄의 핵심이다.

**교훈 2.** AI 인프라는 누적된다. 한 달 전에 깐 모델이 오늘 다른 작업 중에 부산물로 발견되는 모먼트는 흔하다. 청소만 능사가 아니라, 발견한 잔여물을 **별 메모리 줄로 격리**해두는 게 다음에 또 헷갈리지 않게 한다.

## 3막 — 추측과 실측의 갭 (16:11)

노트북에 무슨 모델이 깔려 있는지 추측이 필요했다.

추측: 노트북은 6GB GPU. 셋업 스크립트 `setup-local-llm.ps1` 의 자동 분기 로직은 VRAM ≥ 7.5GB 면 llama3.1, 미만이면 qwen2.5 fallback. 따라서 노트북엔 qwen2.5 가 깔렸을 것이다. **합리적 추측이었다.**

실측: 노트북에 `llama3.1:8b` 가 깔려 있었다. 4.92GB 모델이 6GB VRAM 에 풀로드 된 상태. 빡빡하지만 동작 OK. 콜드스타트 7.83초, 워밍 후 19.6 tok/s.

→ "VRAM 빡빡하면 fallback 이 합리적" 은 일반론적으로 맞지만, **강대종님은 의도적으로 llama3.1 로 통일했다** (모델 검증 일관성 위해). 셋업 스크립트의 자동 분기 디폴트가 그 결정을 이긴 적 없다.

**교훈 3.** 스크립트의 디폴트 동작과 실제 운영 결정은 다른 레이어다. 코드만 보고 운영 상태를 추론하면 자주 틀린다. 실측 한 번이 추측 다섯 번을 이긴다.

## 4막 — 토폴로지 정정 (16:16)

처음에 trend-curator 코드 한 줄 보고 토폴로지를 잘못 그렸다.

추측한 토폴로지: `Mac → 노트북 (1-hop SSH)` — 트리거가 Mac 에서 노트북으로 직접.

강대종님 정정:

> "맥 → 본진, 그리고 본진에서 데스크탑 노트북 이렇게야"

코드 헤더를 다시 읽어보니 명시되어 있었다:

```python
# Hops: Mac -> WSL master -> notebook
# (so SSH key footprint stays minimal)
```

내가 안 읽고 추측했다. 실제 토폴로지는 **2-hop**: Mac → WSL 본진(허브) → 노트북·데스크탑(외곽). 이유는 한 줄짜리 주석에 적혀 있었다 — "SSH 키 풋프린트 최소화."

이게 별 거 아닌 것 같지만 노드가 늘어날수록 가치가 커진다. 외곽 노드 2개에 본진 키 한 세트만 등록하면 된다. Mac 은 본진까지만. 새 노드 합류할 때마다 N×M 키 쌍 등록이 필요 없다.

**교훈 4.** 토폴로지는 그림 한 장으로 머리에 새겨진다. 그 그림 그리기 전에 코드 헤더부터 읽는다.

## 5막 — 기자와 통역사 (16:22~16:27)

다음 갈림길.

> "트렌드큐레이터는 노트북이 아니라 데스크탑에 돌리는거 아니었어?"

코드를 확인하니 환경변수 `NOTEBOOK_HOST="100.89.67.22"` (노트북 IP) 로 하드코딩돼 있었다. 데스크탑 변종 스크립트는 없었다.

근데 이상했다:
- trend-curator 매일 10:00 KST 실행 (낮, "밤 배치" 아님)
- 데스크탑 75 tok/s 가 노트북 22 tok/s 보다 3.4배 빠름
- 누가 봐도 데스크탑이 더 어울림

대안 두 개가 떠올랐다:

- **옵션 A**: 트렌드큐레이터 자체를 데스크탑으로 옮긴다. 데스크탑에 git clone, .env 안전 이전, launchd 잡 수정. 5단계 작업.
- **옵션 B**: 실행은 노트북에서 하되, **LLM 호출만 데스크탑 ollama 로 보낸다.** ollama URL 환경변수 한 줄 변경.

옵션 B 를 추천했다. 강대종님 답:

> "b 내가 이해가 잘안가 쉽게 설명해줘"

여기서 깨달은 게 있다. 엔지니어링 의사결정의 진짜 작업은 옵션 A·B 의 트레이드오프 비교가 아니다. **그걸 사용자가 직관적으로 동의할 수 있게 비유로 풀어내는 것**이다.

비유 한 줄:

> "기자는 노트북이고 통역사는 데스크탑이에요.
> 기자가 자기 자리에서 일하면서 필요할 때 통역사한테 전화 거는 것.
> 기자를 통역사 자리로 굳이 옮길 필요 없어요."

이 한 줄로 결정 났다. 옵션 B.

**교훈 5.** 대안 비교를 끝낸 다음에 일이 끝나지 않는다. 그 다음 **비유로 풀어내기** 가 의사결정의 진짜 마지막 단계다. AI 어시스턴트가 잘하는 영역이기도 하다 (방대한 비유 후보 중 그 사용자에게 맞는 한 줄 고르기).

`fail-soft` 가 핵심이다. LLM 호출 실패해도 다음 아이템 계속, 실패한 자리엔 `(요약 실패)` 표기. 한 아이템 실패가 전체 큐레이션을 죽이지 않는다.

WSL 본진에서 검증 — 2 items / 6.8s. 출력 일부:

> 이 저장소는 코드베이스 전체에 대한 심도 있는 파일을 인용한 기술 부채 ауд이트를 생성하는 클로드 코드 스킬입니다.

> 클라우드 코드(Claude Code), 코덱스(Codex), 항중력(Antigravity), 오픈코드(OpenCode)와 같은 에이전트에 대한 크로스-에이전트 세션 재개 스キ야를 제공합니다.

문제 셋이 한눈에 보인다:

- `ауд이트` ← 키릴 문자가 한국어에 섞임 ("аудит"의 잔여)
- `스キ야` ← 카나 문자 섞임 ("スキーマ"에서 왔을 것)
- `항중력` ← Antigravity 의 뜻을 잡으려고 직역 시도, 실패

**8B 양자화 모델의 토큰화 한계가 그대로 노출**됐다. 의미는 통하지만 거슬린다. README 의 "알려진 한계" 절에 정직하게 추가했다.

**교훈 6.** 로컬 LLM 의 진짜 한계는 "느려서" 가 아니라 **"가끔 키릴 섞인 한국어 뱉어서"** 다. 양자화·토큰화·다국어 학습 데이터 비율 모두에 걸친 깊은 한계라 단순 프롬프트 튜닝으로 안 잡힌다. 이 거슬림을 받아들이거나, 더 큰 모델 (70B?) 로 가거나, 한국어는 클라우드 모델에 보내거나 — 셋 중 결정이 필요하다.

## 7막 — 마지막 1m, 인간이 친다 (16:40)

GitHub 에 push 까지 됐다. 노트북에 git pull 만 하면 끝이었다.

근데 막혔다.

노트북은 HTTPS origin 으로 클론돼 있었고, GitHub 인증 캐시가 없었다. SSH 세션에서 git pull 을 치면 username·password 프롬프트가 뜨는데, 비대화형 세션이라 그 자리에서 **hang** 됐다.

자동화로 해결할 옵션을 찾아봤다:

- scp 로 직접 파일 덮어쓰기 → 시스템 가드 ("공유 인프라 무단 수정 아님")
- ssh-keygen + GitHub deploy key 등록 → 로그인 세션 필요
- gh auth login → 마찬가지로 로그인 세션 필요

**원격에서 자동화로 해결 불가**. 강대종님이 직접 노트북 앞에 가서 한 번 손쓰는 게 답이었다.

이게 자동화의 마지막 1m 라고 부르는 패턴이다. 99% 까지는 에이전트가 한다. 마지막 1m — 인증 캐시 한 번, 토큰 한 번, 권한 한 번 — 은 인간이 친다.

**교훈 7.** 에이전트가 모든 걸 해주지 않는다. 마지막 git pull 한 번이 사람의 손을 부른다. 자동화 비율 100% 가 목표가 되면 안 된다 — **99% 자동화 + 1% 사람의 한 줄** 이 더 안정적이다.

## 메타 — 두 세션을 동시에 굴리면 생기는 것

이 평행본의 제목은 사실 **"같은 시간 옆자리에서는 코드를 짜고 있었다"** 다. 하지만 실제 메시지는 더 단순하다:

**1인 개발자가 Claude Code 두 대를 동시에 굴리면, 두 세션은 서로 모른 채 평행하게 일하다, 사용자가 양쪽을 보면서 합류시킨다.** 그게 비효율이 아니라 분업이다. 두 세션이 같은 일을 안 한다는 게 핵심이다.

오늘의 분업:
- **Mac 본진** — 의사결정 (라파5 vs N100 vs M1 맥미니, 5막 드라마)
- **WSL 본진** — 구현 (trend-curator Phase 2, 7막 디버깅)

두 흐름이 16:53 한 줄로 합류했다. 합류는 자동이 아니다 — 강대종님이 직접 한 거다. AI 어시스턴트는 평행 우주 안에서만 일관적이다. 평행 우주 사이를 잇는 건 사람의 일이다.

**Ep.3 메인 본의 교훈** — "사기 전에 집 한 번 둘러보세요."
**Ep.3 평행본의 교훈** — **"두 세션을 동시에 굴릴 때, 두 세션이 같은 일을 하지 않게 하세요."**

같은 사용자, 같은 시간, 다른 일. 이게 1인 인디 개발자가 두 비서를 굴리는 자세다.

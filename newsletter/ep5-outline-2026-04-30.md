## 막 구조

### 1막 — 합격 점수표가 있었다

- 약먹자·더치페이 1.0 출시 사이클 (4/24~4/28) 이 먼저 있었다. 4일 안에 제출→거절→Resolution Center 답글→승인까지 끝났다.
- 거절 사유는 Guideline 2.1 — Information Needed. 6항목 정보 요구 (demo video / App Purpose / Main Features / External Services / Regional / Regulated Industry).
- 답글 송신 후 ~36h 안에 Apple 이 매뉴얼로 큐를 다시 돌리고 승인했다.
- 그 점수표를 그대로 한줄일기에 적용했다. 4/28 23:04 KST 제출 → 24~48h 결과 메일 예상.
- 같은 동물이 아니었다.

### 2막 — China Deep Synthesis

- 4/29 06:43 KST Apple 메일 도착: **UNRESOLVED_ISSUES**, Guideline 5 — Legal.
- 4/30 11:00 KST 강대종님 ASC 캡처로 사유 확정: AI 한줄응원 기능이 중국 본토의 "Administrative Provisions on Deep Synthesis of Internet-based Information Services" 적용 대상이라는 판정. 즉 **법규 거절** 이지 정보 요청이 아님.
- 해결안 두 갈래:
  1. 중국 본토 법률 자문을 받아 응답 (시간·비용 큼)
  2. **중국 본토를 출시국에서 제외** (즉시 가능, 174/175 territory 유지)
- 결정은 (2). 4/29 08:05 KST Resolution Center 답글 송신 — "We have removed China mainland from this app's availability per Guideline 5 - Legal feedback. Please re-review."
- 약먹자 사이클의 학습("답글 보내면 큐가 다시 돌아간다")을 그대로 따랐다.

### 3막 — Apple 이 답을 안 했다

- 약먹자·더치페이는 답글 보낸 지 ~36h 만에 큐가 다시 돌아갔다. 한줄일기는 26h 가 지났는데 reviewSubmission state 가 그대로 `UNRESOLVED_ISSUES` 였다.
- ASC API 로 들여다보니 옛 reviewSubmission 이 appStoreVersion 을 점유한 채로 잠겨있었고, 새 reviewSubmission 한 개가 자동 생성돼있긴 했지만 **items=0 빈 껍데기** 였다.
- 정공법으로 풀려고 시도했더니 모두 막혀있었다:
  - POST 새 sub 에 appStoreVersion attach → `409 ITEM_PART_OF_ANOTHER_SUBMISSION`
  - DELETE 옛 sub 의 reviewSubmissionItem → `409 Item was already submitted`
- 즉 **Resolution Center 답글이 자동으로 큐를 회전시키는 거절 클래스 (Guideline 2.1) 와, 안 회전시키는 클래스 (Guideline 5 Legal) 가 다르다.** 약먹자 사이클의 점수표는 후자에 적용되지 않는다.

### 4막 — PATCH canceled=true

- 우회 경로 4단계, 모두 ASC API:

```python
# 1. 옛 sub 를 우리 손으로 취소
PATCH /v1/reviewSubmissions/7469efbf
  { "canceled": true }
# state: WAITING_FOR_REVIEW → CANCELING → COMPLETE  (~30s 폴링)

# 2. 새 sub 에 appStoreVersion 붙이기
POST /v1/reviewSubmissionItems
  { "submission": "e58db9c9", "appStoreVersion": <id> }
# 201 OK  (옛 sub 가 풀렸으니 점유 해제)

# 3. 새 sub 제출
PATCH /v1/reviewSubmissions/e58db9c9
  { "submitted": true }
# 200 OK, state = WAITING_FOR_REVIEW
# submittedDate = 2026-04-30T02:06:57Z (KST 11:06)
```

- 핵심 발견: **잠긴 reviewSubmission 은 PATCH `canceled: true` 로 우리가 직접 풀 수 있다.** Apple 매뉴얼 액션을 기다릴 필요 없다.
- 어제(4/29) 세션이 막혔던 이유는 이 잠금의 존재 자체를 발견 못 했기 때문. 정공법 두 번 시도 (POST attach / DELETE item) 가 둘 다 409 로 같은 잠금을 다른 각도에서 본 신호였는데, 같은 잠금이라는 인식이 늦었다.

### 5막 — 2시간 23분

- 11:06 KST 재제출 → **13:29 KST Apple 승인 메일** 도착. "Review of your 한줄일기 - AI 한줄응원 (iOS) submission is complete."
- 턴어라운드 약 **2시간 23분**. 보통 24~48h 큐 기준으로 이례적으로 빠름. 같은 빌드를 같은 메타데이터로 재제출했고 차이는 출시국 1개 (CHN) 뿐이었으니, reviewer 입장에선 "전 거절 사유 해결 확인 → pass" 한 번으로 끝났을 가능성.
- ASC 검증: appStoreState `READY_FOR_SALE`, releaseType `AFTER_APPROVAL` → 승인 즉시 자동 출시. ₩1,900 KOR 유료. App Store URL https://apps.apple.com/kr/app/id6764308678
- 출시 사이클 총합:
  - 4/28 23:04 첫 제출 → 4/29 06:43 거절 (~7h)
  - 4/29 06:43 거절 → 4/30 11:06 재제출 (~28h, 그중 26h 는 Resolution Center 답글 대기)
  - 4/30 11:06 재제출 → 13:29 승인 (2h23m)
- 약먹자/더치페이 4일 사이클과 비교하면 한줄일기는 약 38시간. 더 짧다. **이유는 자동화가 두 번째 시도에서 작동했기 때문이다.**

> 거절은 한 번이지만, 다시 돌리는 건 매번이다.

## 핵심 메시지

자동화의 진짜 가치는 첫 제출 성공률이 아니라 **거절 받고 두 번째 시도까지의 손 거리** 다. 약먹자 사이클 (Guideline 2.1) 은 답글 한 통으로 자동 회전됐지만, 한줄일기 사이클 (Guideline 5 Legal) 은 답글이 무시되고 큐가 잠겼다. 같은 답글 패턴을 두 번째에도 자동으로 보낼 수 있어야, 거절 클래스가 다르다는 사실을 빠르게 알아차리고 (정공법 두 번 막힘 → 우회) 4단계 ASC API 호출로 풀 수 있다.

Ep.4 가 "이미 한 결정이 무너지지 않게 하는 어려움" 이었다면 Ep.5 는 **"이미 보낸 결정이 거절당했을 때 다시 보내는 어려움"** 이다.

## 동사 후보 (3개)

본문 작성 시 강대종님이 픽:

1. **돌리기 (return-to-queue)** — 거절을 다시 큐에 돌렸다. 시리즈 동사들과 어조 매칭, 운반체 모티프 일관. Ep.4 "잇기" 의 후속으로 자연스러움. *추천 1순위.*
2. **밀기 (push-through)** — 잠긴 reviewSubmission 을 손으로 취소해서 큐에 강제로 밀어넣었다. 4막의 PATCH canceled=true 동작 묘사로 강함. 단, 시리즈 다른 동사가 다 "수동·결정" 톤인데 "밀기" 는 강압이라 살짝 빠짐.
3. **버티기 (endure-rejection)** — 4/24 시작해서 6일 동안 거절·재제출·승인 사이클을 버틴 결과. 인내 톤, 약먹자 4일 대비 한줄일기 6일 비교 가능. 단, 사실 자동화 덕에 "버틴" 시간이 짧아서 메시지와 살짝 어긋남.

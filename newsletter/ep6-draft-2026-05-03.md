## 문제는 '잊어버림'이었다

앱이 하나일 때는 괜찮았다. 수동 빌드하고, Xcode 열고, Transporter 실행하고, ASC 에서 심사 제출 버튼 누르면 됐다. 한 번에 30분.

앱이 네 개가 되자 달라졌다.

메모요, 한줄일기, 약먹자, 더치페이. 각각 다른 스케줄로 업데이트가 나오고, 각각 다른 심사 상태를 가졌다. 어떤 날은 두 앱을 동시에 제출해야 했다. 어떤 날은 이미 심사 중인 앱이 있어서 다른 앱 제출 타이밍을 조율해야 했다.

제일 큰 문제는 **잊어버림** 이었다. 코드 머지하고 잠들면, 다음 날 아침에야 "아, 어제 빌드 안 했지" 가 떠올랐다.

---

## night-builder v2: 자는 동안 빌드

해결책은 간단했다. 빌드를 밤에 자동으로 돌리면 된다.

Mac mini 에 launchd 잡을 심었다. 매일 새벽 3시, 4개 앱을 순서대로:

1. `git pull` — 최신 코드 받기
2. `flutter build ipa` — iOS 빌드
3. `flutter build appbundle` — Android 빌드
4. 결과를 텔레그램으로 보고

새벽 3시에 시작하면 4시쯤 끝났다. 아침에 일어나면 텔레그램에 "PASS 4 / FAIL 0" 이 와 있었다.

중요한 설계 결정 하나: **FAIL 시 자동 재시도 없음.** 실패하면 보고만 하고 멈춘다. 자동 재시도는 실패 원인을 묻어버린다. 사람이 원인 보고 다음 날 다시 돌리는 게 낫다.

## asc-deliver: 빌드에서 심사 제출까지

빌드까지 자동화했는데, 업로드와 심사 제출은 여전히 수동이었다.

ASC API 에 `asc-deliver` 래퍼를 만들었다. 플로우는:

```
ipa 파일 존재 확인
→ altool 로 App Store Connect 업로드
→ 업로드 완료 대기 (보통 5-10분)
→ reviewSubmission 생성
→ 심사 제출 완료
```

주의점이 있었다. ASC API 는 App Privacy, Pricing 설정을 못 건드린다. 신규 앱 첫 제출 때 이 두 화면은 Playwright 로 직접 클릭해야 했다. 기존 앱 업데이트라면 건너뛸 수 있다.

또 하나: **심사 제출 버튼은 API 로 누를 수 없다.** reviewSubmission 을 만들고 submit 하는 API 는 있지만, 실제 Xcode Organizer 나 ASC 웹에서 하는 "제출" 과 다른 경로다. 헷갈렸던 부분.

결국 asc-deliver 가 커버하는 범위: 업로드 + reviewSubmission 생성까지. "제출" 버튼은 ASC 웹에서 한 번 수동으로.

## 실제로 어떻게 됐나

한줄일기 1.1.0 Android 업데이트가 처음 풀자동으로 돌아갔다.

밤 3시 night-builder 가 빌드 시작 → 새벽 4시 aab 파일 생성 → asc-deliver 가 Play Console 업로드 → 내부 테스트 트랙 배포 완료. 내가 한 건 자는 것.

아침에 텔레그램 확인했더니:

```
빌드: PASS
업로드: PASS
트랙: internal
```

직접 Play Console 들어가서 보니 진짜로 올라가 있었다.

## 자동화의 진짜 비용

자동화를 구축하는 데 약 이틀이 들었다. 이틀 치 수동 작업을 자동화하는 데 이틀을 쓴 셈이다.

손익분기점: 앱이 4개, 월 1~2회 업데이트라면 월 8회 수동 작업. 한 번에 30분이면 월 4시간. 1년이면 48시간.

자동화 구축 비용이 이틀(16시간)이면, 4개월이면 본전이다.

하지만 진짜 이득은 시간이 아니었다. **잊어버리지 않는다는 것** 이었다. 코드가 main 에 머지되면 그날 밤 자동으로 빌드된다. 내 기억에 의존하지 않는다.

---

## 1인 개발자에게 자동화가 필요한 이유

팀이 있으면 체크리스트를 서로 확인한다. 혼자면 그게 없다.

자동화는 혼자일 때 팀을 대신한다. 잠든 사이 빌드가 돌고, 업로드가 되고, 텔레그램에 보고가 온다. 아침에 일어나면 어젯밤에 무슨 일이 있었는지 알 수 있다.

다음 번에 앱 다섯 번째를 만든다면, 빌드 자동화 세팅은 첫 번째 작업이 될 것이다.

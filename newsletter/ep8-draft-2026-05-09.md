# 코레일 매크로 도전기 — Playwright 로 기차표를 잡으려 했다

_바이브코딩 뉴스레터 Ep.8 초안_

_2026-05-09_

---

> 명절 기차표가 열리는 시간은 새벽 6시다.
>
> 손으로 새로고침을 누르면 이미 늦다. 그래서 나는 매크로를 만들기로 했다.

개발자가 되면 한 번쯤 이런 생각을 한다. "이 반복 작업, 코드로 짜면 나 안 해도 되는데." 나는 그 순간이 코레일 예매 창 앞에서 왔다.

---

## 시작은 단순했다

계획은 간단했다. Playwright 로 코레일 사이트 접속 → 로그인 → 열차 검색 → 잔여석 있으면 바로 예매 버튼 클릭. 10분이면 짜겠다고 생각했다.

실제로는 나흘이 걸렸다.

첫 번째 장벽은 **로그인** 이었다. 아이디/비밀번호 입력 폼은 Playwright 로 쉽게 채울 수 있었다. 문제는 그 다음. 로그인 버튼을 누르면 특정 조건에서 캡차가 뜬다. 그냥 image captcha 도 아니고 텍스트를 그림으로 변환해 놓은 방식이라 OCR 로 읽기도 힘들었다. 결국 첫날은 여기서 막혔다.

---

## 두 번째 함정: 세션 만료

캡차를 사람이 직접 풀고 세션을 유지한 채로 매크로에 넘겨주는 방식으로 우회했다. 쿠키를 파일로 저장하고, 다음 실행 때 불러오는 전략이다.

문제는 코레일 세션이 예상보다 훨씬 빨리 만료된다는 것. 일반 쇼핑몰은 30분~1시간이지만, 코레일은 특정 페이지에서 이동하면 세션이 끊겨버렸다. "로그인 했는데 왜 다시 로그인 화면이 나와?" 를 열 번쯤 반복하고 나서야 패턴을 파악했다.

세션 유지 상태를 확인하는 코드를 먼저 짜고, 만료됐으면 재로그인하는 로직을 붙였다. 그러자 이번엔 캡차가 또 걸렸다.

---

## 세 번째 함정: DPI 스케일 좌표 어긋남

윈도우에서 돌리는 Python + Playwright 조합이었다. 맥에서 개발하고 윈도우에서 실행하니 화면 해상도 설정이 달랐다.

Windows 디스플레이 배율이 150%로 설정돼 있었다. Playwright 는 논리 좌표를 쓰는데, pynput(마우스 자동 클릭 라이브러리) 은 물리 좌표를 쓴다. 두 좌표계가 서로 다른 배율로 작동하니까 클릭 위치가 전혀 딴 데로 가버렸다.

한참 헤매다 원인을 찾았다. `SetProcessDpiAwareness(2)` — 프로세스 진입점 상단에 이 한 줄을 추가하고 재시작하면 DPI 배율을 물리 화면 기준으로 통일해준다. 파이썬 코드 맨 위에 ctypes 로 윈도우 API 호출 한 줄. 이걸 모르면 클릭이 계속 어긋난다.

```python
import ctypes
ctypes.windll.shcore.SetProcessDpiAwareness(2)
```

## 결국 어떻게 됐나

완전 자동화는 실패했다. 캡차 때문이다. 캡차를 사람이 한 번 풀고 세션을 넘겨주는 반자동 방식까지는 됐다.

세션 넘겨받은 이후 흐름은 이렇게 됐다:

1. 새벽 5시 55분, 로그인해서 쿠키 저장
2. 6시에 매크로 실행 — 열차 조회 + 잔여석 폴링
3. 잔여석 생기면 자동 예매 버튼 클릭
4. 결과 텔레그램 알림

실제로 이렇게 해서 표를 잡은 적이 있다. 캡차 한 번 풀고 자리에 앉아있으니까, 6시 1분에 "예매 완료" 텔레그램 메시지가 왔다.

전부 자동화는 아니었지만, "새로고침을 손으로 200번 누르는" 것보다는 훨씬 나았다.

---

## 배운 것

기술적 함정보다 더 중요했던 교훈: **자동화에는 항상 사람이 비집고 들어오는 구간이 있다.** 캡차는 사람이 풀어야 한다는 전제 위에 설계된 것이다. 그 구간을 자동화로 없애려 할수록 더 많은 우회로가 필요해지고, 코드는 점점 복잡해진다.

적당한 선에서 반자동화로 멈추는 것도 엔지니어링이다.

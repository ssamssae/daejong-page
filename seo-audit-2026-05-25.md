## 우선순위별 권고
### P0 (즉시)
- `sitemap.xml` 생성: 감사 대상 30개 페이지를 `https://work.kangdaejong.com/...` 절대 URL로 포함한다. 개인정보 처리방침 페이지까지 넣되, 중요도는 홈/포트폴리오/콘텐츠 허브보다 낮게 둔다.
- `robots.txt` 생성: 최소 `User-agent: *`, `Allow: /`, `Sitemap: https://work.kangdaejong.com/sitemap.xml`을 선언한다.
- 모든 감사 대상 페이지에 canonical 추가: `index.html`은 `https://work.kangdaejong.com/`, 나머지는 파일 경로 기반 절대 URL로 지정한다.
- `index.html`에 h1과 meta description 추가: 홈은 브랜드 검색과 사이트 이해의 기준점이다. 예: "강대종(@ssamssae)의 1인 앱 개발 포트폴리오, Claude Code 기반 작업일지, 뉴스레터, 인사이트와 출시 앱을 모은 개인 사이트."
- 공통 OG/Twitter 템플릿 추가: 최소 `og:title`, `og:description`, `og:image`, `og:type`, `og:url`, `og:site_name`, `twitter:card`, `twitter:title`, `twitter:description`, `twitter:image`를 모든 핵심 페이지에 넣는다.

### P1 (이번 주)
- `Person` JSON-LD를 홈에 추가하고, 공통으로 재사용한다. `name: 강대종`, `alternateName: ssamssae`, `url: https://work.kangdaejong.com/`, `sameAs`에 실제 공개 SNS/프로필을 넣는 구조가 적합하다.
- `WebSite` JSON-LD를 홈에 추가한다. 사이트명은 `강대종` 또는 `강대종 - 1인 개발 포트폴리오`로 통일한다.
- 콘텐츠형 페이지에 구조화 데이터 분화:
  - `insight.html`, `issue.html`: 동적 상세 페이지라면 JS 렌더링 데이터 기준으로 `Article` 또는 `TechArticle` JSON-LD를 생성한다.
  - `insights.html`, `issues.html`, `worklog.html`, `newsletter.html`: 목록 페이지는 `CollectionPage` + `BreadcrumbList`가 적합하다.
  - `portfolio.html`, `vrl-benchmark.html`: `ProfilePage`, `CreativeWork`, `SoftwareApplication` 후보를 검토한다.
- description을 페이지별로 120~160자 권장 범위에 맞춰 확장한다. 현재 존재하는 description 17개도 대부분 34~71자로 짧다.
- `portfolio.html`, `stack.html`, `timeline.html` 등 description 없는 주요 비개인정보 페이지부터 보강한다.
- `privacy.html`과 `privacy-memoyo.html` 중복 title/내용 정책을 정한다. 하나를 legacy URL로 둘 경우 canonical을 대표 URL로 보내야 한다.

### P2 (백로그)
- 공통 head partial 또는 빌드 스크립트 도입을 검토한다. 현재 정적 HTML이 개별 파일로 흩어져 있어 canonical, OG, favicon, JSON-LD 누락이 반복될 가능성이 높다.
- 대표 공유 이미지(`og:image`)를 만든다. 홈/포트폴리오/뉴스레터/인사이트/이슈 정도는 각기 다른 이미지가 좋고, 나머지는 공통 이미지로 시작해도 충분하다.
- favicon 정책 정리: `homepage_icon.svg`를 공통 `rel="icon"`으로 적용하고, PNG fallback 및 `apple-touch-icon`을 추가한다.
- `hreflang`은 현재 단일 한국어 사이트라 보류해도 된다. 영문 소개나 앱 랜딩 페이지가 생기면 `ko`, `en`, `x-default`를 같이 설계한다.
- 동적 렌더링 페이지의 SEO 한계를 점검한다. `insight.html`, `issue.html`처럼 상세 콘텐츠가 JS로 채워지는 페이지는 검색엔진이 실제 상세 본문과 title/description을 안정적으로 보지 못할 수 있다.

## 추가 발견 / 보너스
- 감사 대상 30개 중 개인정보 처리방침이 9개다. 이들은 검색 유입 목적보다 신뢰/심사 목적이 크므로 canonical, noindex 여부, sitemap 포함 여부를 의도적으로 정하면 좋다. 앱 심사용 URL이면 sitemap 포함은 무방하지만 우선순위는 낮다.
- `sise.html`만 favicon 링크가 있다. 공통 head 정책이 없다는 강한 신호다.
- `index.html`의 이미지 alt는 모두 존재한다. 이 부분은 좋다.
- `portfolio.html`과 `vrl-benchmark.html`은 JS 문자열로 이미지를 생성하는 부분이 있어 정적 SEO 감사에서 실제 이미지 수가 낮게 잡힌다. 검색엔진/공유 봇 관점에서는 정적 HTML에 핵심 이미지와 텍스트가 더 많이 남는 편이 유리하다.
- 제목은 대부분 고유하지만, 개인정보 처리방침 계열은 브랜드/앱명을 title 앞에 넣는 일관된 패턴이 필요하다.
- 권장 구현 순서: sitemap/robots → canonical → 홈 h1/description → 공통 OG/Twitter/favicon → Person/WebSite JSON-LD → 주요 콘텐츠 description 확장.

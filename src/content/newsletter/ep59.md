*두 저장소를 동시에 올리는 작업이 돌고 있었다. 더치페이 앱 쪽이 자격증명 문제로 실패했다는 메시지가 떴다. 옆에는 "Cancelled: parallel tool call errored"라고 적혀 있었다. 페이지 쪽도 취소됐다는 뜻이었다. "그럼 페이지는 처음부터 다시 하면 되는데, push만 다시 넣으면 되겠지" — 그 순간 머릿속에서 그렇게 흘러갔다.*

---

나는 요즘 여러 저장소를 동시에 관리한다. 앱도 있고, 뉴스레터 페이지도 있고, 관련 설정 파일들도 있다. 작업이 겹치는 날은 여러 저장소에 동시에 코드를 올려야 할 때가 있다. 자동화된 환경에서 이걸 한꺼번에 처리하면 시간이 절반으로 줄어든다. 명령을 동시에 두 개 발사하고, 둘 다 성공 신호를 기다리면 된다 — 이론적으로는.

git push라는 말을 잠깐 풀어두자. 코드를 수정하면 먼저 "이 파일을 기록 대상으로 올린다"(add)는 동작을 하고, "지금까지 바꾼 것들을 하나의 묶음으로 저장한다"(commit)는 동작을 한 뒤, 마지막으로 "이 묶음을 원격 서버에 업로드한다"(push)하는 순서로 진행된다. 이 세 단계가 붙어있는 게 보통의 코드 올리기다.

## 한쪽이 실패했을 때 다른 쪽은 어디에 있는가

그날 사고의 핵심은 이 질문에서 왔다. 두 저장소를 동시에 올리는 명령을 발사했다. 더치페이 앱 저장소 쪽에서 자격증명이 없어 push가 실패했다 — HTTPS 방식의 원격 주소를 쓰고 있었는데, 비대화형 환경에서는 ID와 비밀번호를 입력할 창이 뜨지 않아서 그냥 실패하는 상황이었다.

그 순간 옆에 있던 페이지 저장소 쪽 결과에 "Cancelled"가 찍혔다. 병렬로 돌리던 작업 중 하나가 에러를 내면, 함께 돌던 다른 쪽이 자동으로 취소되는 방식이었다. 당연한 동작이다. 그런데 나는 그 "취소"가 정확히 어디서 일어난 건지 확인하지 않았다.

머릿속에서 이렇게 정리됐다. "더치페이 쪽은 push에서 실패했다. 페이지 쪽은 취소됐다. 페이지는 add랑 commit은 다 됐을 수 있으니까 push만 다시 하면 되겠다." 그렇게 판단했다.

## 취소는 '멈춤'이 아니라 '처음부터 0건'이다

재시도를 돌렸다. 페이지 저장소에 git push만 다시 넣었다. 명령은 성공했다는 신호를 보내왔다. 그 결과를 보고서에 담아 올렸다. 작업 끝.

그런데 실제로 원격 서버에 올라간 건 내용 없는 빈 브랜치였다. add도 commit도 없이 push만 한 것이었다. commit이 없는 브랜치를 push하면 브랜치 자체는 생기지만, 그 안에 아무것도 없다. 서버에는 "이 브랜치가 있습니다"라는 기록만 남고, 그 브랜치에 담겨야 했던 변경 내용은 없다.

왜 그렇게 됐냐면 — 병렬로 돌다가 취소된 쪽은 처음 명령부터 0건 수행된 것이다. add도, commit도 실행되지 않은 채 취소된 거다. 그러니 push만 다시 돌리면 이전 commit 상태 그대로 push가 된다. 새로 만들려던 변경 사항은 어디에도 없다.

"취소됐다 = 어디선가 멈췄다"가 아니라 "취소됐다 = 처음부터 아무것도 실행 안 됐다"가 정확한 해석이었다.

## 보고서에 내용이 있었다는 게 증거가 아니었다

사고를 더 오래 끌게 만든 건 보고서 때문이었다. 작업 결과를 정리해 올리는 보고서 안에 페이지 쪽 spec 본문이 들어가 있었다. 그 내용을 보면서 "올라갔겠지" 하고 느꼈을 것이다. 보고서에 내용이 있으니 push가 됐다고 연결 지은 것이다.

그런데 보고서 본문에 내용이 있다는 건, git push가 됐다는 증거가 아니다. 두 가지는 아무 관계가 없다. 보고서는 그냥 텍스트를 담아 보낸 거고, 서버에 올라간 브랜치와는 별개다. "보고서에 있다 = 서버에 있다"고 읽은 것 자체가 틀렸다.

이게 자동화 환경에서 자주 빠지는 함정이기도 하다. 결과처럼 보이는 것들이 중간 산출물인데, 그걸 최종 결과로 착각하는 것. 텍스트가 있다, 파일이 있다, 로그가 찍혔다 — 그게 다 "됐다"는 뜻은 아니다.

## 병렬 묶음에서 에러 옆에 취소가 뜨면

그날 이후로 규칙 하나를 박았다. 병렬로 돌린 묶음 명령(add+commit+push)에서 한쪽에 에러가 뜨고 다른 쪽에 "Cancelled"가 뜨면 — 취소된 쪽은 첫 명령부터 다시 실행한다. push만 다시 하지 않는다. add부터 다시 한다.

그리고 push 명령이 성공 신호를 보내왔다고 해서 끝이 아니다. 두 단계를 더 확인한다. `git log --oneline -1` 으로 방금 만들어진 commit의 식별자(SHA)와 메시지를 본다. 그리고 `git rev-list --count origin/<브랜치>..HEAD` 명령으로 "로컬에는 있는데 서버에는 아직 안 올라간 commit이 0개인가"를 확인한다. 이 숫자가 0이 아니면 push가 제대로 안 된 거다.

이 두 줄짜리 검증을 push 직후 확인하는 걸 습관으로 만들었다. 성공 신호 하나만 믿고 끝내던 것에서, 실제 반영 결과를 눈으로 보고 끝내는 것으로.

## 가장 흔한 건 전부 성공도, 전부 실패도 아니다

병렬 작업을 오래 돌리다 보면 이런 가정을 하게 된다. "성공이면 둘 다 됐고, 실패면 둘 다 안 됐겠지." 그게 자연스러운 느낌이다. 동시에 출발했으니까.

그런데 실제로 가장 많이 일어나는 건 부분 성공이다. 하나는 완전히 됐고, 하나는 절반쯤 됐거나 아예 0건이거나. 에러 종류에 따라, 명령이 어디서 멈췄냐에 따라, 다음 시도에서 어디서부터 다시 해야 하는지가 완전히 달라진다.

그러니까 "한쪽이 취소됐다"는 신호를 보면 그냥 다시 시작하는 게 맞다. "어디까지 됐을까" 추측하고 거기서부터 이어가려는 시도가, 오히려 새로운 문제를 만든다. 추측 대신 재실행. 묶음은 묶음 단위로.

## 배운 것

자동화가 많아질수록 "에러 옆의 취소"를 자주 보게 된다. 그걸 볼 때마다 이제는 먼저 묻는다 — 저쪽은 어디서 멈춘 게 아니라 아예 시작을 못 한 거 아닌가?

push 성공 신호를 믿지 않는다. `git log --oneline -1`로 commit이 실제로 있는지, `rev-list --count`로 서버와 차이가 없는지 — 이 두 줄을 확인하고 나서야 "됐다"고 쓴다.

그리고 보고서에 내용이 있다는 걸, 서버에 반영됐다는 증거로 읽지 않는다. 그 두 가지는 아무 관계가 없다.

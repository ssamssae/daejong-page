6월 21일 새벽은 전날 만든 운영 규칙을 실제로 쓰는 시간이었다. 긴 task는 쪼개야 했고, Claude 리뷰는 바쁜 노드 대신 idle 노드로 보내야 했고, Codex는 인사이트와 작업일지 발행까지 처리해야 했다. 새 운영면이 문서에서 실행으로 넘어온 셈이다.

## 1. 긴 task는 상태가 갈라지는 순간 쪼갠다

`T-260617-01`은 idempotent-step-guard 우산 task였다. 본진 observe-canary는 이미 시작됐지만, block 모드와 5노드 rollout은 아직 남아 있었다. 한 task 안에 관찰, 구현, 배포, ack 조건이 섞이자 더 이상 읽기 어려워졌다.

그래서 todo skill에 task 쪼개기 기준을 넣었다. 완료조건, 상태, 담당 노드, ack 필요 여부가 갈라지면 정상 신규 ID로 분리한다. `[o]`는 관찰 상태로 정본화했고, Codex adapter manifest도 갱신했다. 5노드에는 Claude todo skill과 Codex adapter가 함께 반영됐다.

## 2. Claude 리뷰 요청은 idle 노드로 보낸다

Claude 리뷰 요청이 본진에서 막히면 수동으로 노트북 Claude에 fallback을 보내야 했다. `T-260621-04`는 이 흐름을 thin CLI로 정리하는 작업이었다. 노트북 Claude 1-pass 리뷰는 조건부 GO를 냈다. 신규 데몬 금지, 명시 호출만 허용, fallback은 최대 1개, task_id dedup, 양쪽 busy면 held 보고.

맥미니는 그 조건에 맞춰 `claude-review-router.sh`와 테스트를 만들었다. PR #160이 열렸고, 이후 본진에서 `ca4c6a1`까지 정리되어 5노드 배포와 테스트가 끝났다.

> **배운 것** — 리뷰 라우팅은 자동 fan-out이 아니라, 실패한 한 요청을 안전하게 한 번 옮기는 문제다.

## 3. 무무 v2는 기술 성공, 품질 실패였다

한줄일기 무무 v2는 데스크탑에서 재학습되고 PR #34로 올라왔다. 맥미니는 아니키 iPhone에 release 빌드를 설치했고, 실제 무무 버튼 재생까지 확인했다. 설치와 실행은 성공이었다.

하지만 청취 결과는 REJECT였다. v2 length_scale 1.3보다 기존 e149 baseline의 품질이 더 좋았다. 그래서 PR #34는 머지하지 않았고, task는 관찰/대기 상태로 내려갔다. 이 판단은 자동 테스트로 대체할 수 없는 영역이었다.

## 4. Codex가 인사이트 발행까지 맡았다

새벽 후반에는 Matt Pocock의 agentic engineering workflow 영상 인사이트를 발행했다. 자막을 추출하고, Astro insight markdown을 만들고, `daejong-page`에 push했다. 발행 커밋은 `ea0dc4e`, glossary hook은 `80bf668`이었다. 라이브 URL은 200으로 확인됐다.

이 작업은 전날 만든 Codex insight adapter의 첫 실전 사용에 가까웠다. Claude skill 원본을 읽고, Codex가 같은 출력 계약으로 발행까지 수행했다.

## 5. 그래서 뭐가 남았나

오늘 남은 것은 세 가지다. 첫째, task 쪼개기와 `[o]` 상태가 5노드에 배포됐다. 둘째, Claude 리뷰 요청 라우터가 테스트와 함께 정리됐다. 셋째, 무무 v2는 기술적으로 설치됐지만 품질 기준에서 보류됐다.

그리고 이 파일 자체도 누락 작업일지 백필 작업의 일부다. 6월 21일은 아직 끝난 날이 아니지만, 새벽까지의 운영 흔적은 이미 충분히 하나의 스냅샷이 됐다. 이후 작업은 다음 버전 파일로 남기면 된다.

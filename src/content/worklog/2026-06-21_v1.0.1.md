6월 21일 후반부는 Codex 메인 운영을 더 단단하게 만드는 시간이었다. 새벽에는 task 쪼개기와 리뷰 라우터를 정리했다면, 이후에는 공개 사이트, 비용 가드, MCP 재연결, 텔레그램 브릿지, 5노드 배포 기준이 한꺼번에 현실 검증을 받았다.

## 1. 공개 운영면을 실제 상태에 맞췄다

오전에는 `daejong-page` 쪽 정리가 이어졌다. 제품 페이지, 로컬 제품 헤더, 10노드 fleet standard, 정책 데이터 재생성, 뉴스레터 ep52/ep53, system 페이지 update가 PR #219부터 #226까지 올라갔다. Codex 200달러, Claude 100달러라는 실제 결제 전제도 공개 운영 설명과 내부 가드에 반영됐다.

이 작업은 단순 문서 정리가 아니었다. 이제 공개 페이지가 실제 운영 구조를 설명해야 했다. Codex는 기본 실행 엔진이고, Claude는 planner/reviewer/gatekeeper 쪽으로 내려간다. 이 전제가 사이트, task, skill, 자동화 문구에 같은 방향으로 박혀야 했다.

## 2. MCP watchdog은 한계를 드러냈다

오후에는 Telegram MCP 끊김 대응이 중심이었다. `telegram-poll-watchdog.sh`는 MCP 일반 재연결 쪽으로 확장됐고, `mcp-reconnect-all-failed.sh`와 PATH helper도 들어갔다. 하지만 실측 결과가 중요했다. Playwright MCP는 상주 프로세스가 아니었고, `pgrep` 기반 생사 판단은 idle 세션에서 오발할 수 있었다. 또 `bun`이나 `npx`가 PATH에 없어서 처음부터 spawn 실패한 경우는 watchdog 재시작으로 해결되지 않았다.

그래서 결론은 더 분명해졌다. MCP를 더 세게 때려서 살리는 방향이 아니라, 텔레그램 브릿지 자체를 MCP 의존에서 분리해야 한다. 아니키가 정정한 것처럼 MCP 문제로 보이는 끊김은 앞으로 Claude Telegram Bridge로 흡수하고, healthy MCP에는 불필요한 주입을 하지 않는 쪽이 맞다.

> **배운 것** — 죽은 프로세스를 다시 띄우는 것과 세션이 실제로 다시 연결되는 것은 같은 문제가 아니다.

## 3. Codex 브릿지는 Codex 전용으로 좁혔다

저녁에는 `telegram-agent-bridge`의 방향을 정리했다. 처음에는 다른 AI도 붙일 수 있는 범용 브릿지처럼 설명했지만, 유지보수 리스크가 컸다. 실제로 아니키가 쓰는 것은 Codex와 Claude 두 축이고, 둘을 한 프로그램 안에서 겸용으로 키우면 작은 수정 하나가 양쪽 런타임을 동시에 꼬이게 만들 수 있었다.

그래서 README와 spec은 Codex 전용으로 좁혔다. 다른 AI 지원 문구를 내리고, Codex REPL bridge로 전문성을 맞췄다. Claude는 Claude 전용 브릿지로 따로 간다. 이 분리는 기능 축소가 아니라 운영 안전장치다.

## 4. Claude Telegram Bridge v1이 들어왔다

밤에는 Claude Telegram Bridge v1 spec이 만들어지고, 이어 PR #163이 머지됐다. 중간에 리뷰 블로커가 있었다. idle 세션 ack, stale injection 방지, retry race, bot inventory, 빌드 검증이 걸렸다. `작업 받음` ack와 10-bot inventory가 추가됐고, round2 race 수정까지 끝난 뒤 `72ec8bd`로 머지됐다.

다만 이것은 daemon을 바로 켠다는 뜻은 아니다. PR 머지는 코드와 구조의 승격이고, canary와 실제 enable은 별도 GO가 필요하다. 오늘은 MCP를 대체할 Claude 브릿지 축이 코드베이스에 들어온 날로 보는 게 정확하다.

## 5. 그래서 뭐가 남았나

오늘 남은 정리는 명확하다. Codex 브릿지는 Codex 전용으로 유지한다. Claude 브릿지는 별도 제품처럼 키운다. MCP는 핵심 통신 경로가 아니라 보조 도구로 낮춘다. 긴 task는 상태가 갈라지면 쪼개고, 끝난 작업은 task 마킹까지 해야 완료다.

새 이슈나 새 메모리로 더 박을 내용은 줄였다. PR #163 블로커는 이미 task와 PR 리뷰 흐름 안에서 처리됐고, MCP 후보들은 "Claude Telegram Bridge로 이동"이라는 결론에 흡수됐다. 오늘의 핵심은 더 많이 자동화한 것이 아니라, 어디를 분리해야 오래 버티는지 확인한 것이다.

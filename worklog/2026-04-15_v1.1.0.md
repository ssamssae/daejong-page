# 2026.04.15 작업일지 v1.1.0 — 양대 스토어에 동시에 밀어넣은 하루

> 저녁, iOS 쪽은 Transporter 로 ipa 를 올려 **`WAITING_FOR_REVIEW`** 까지 찍었다.
> 밤, Android 쪽은 **40MB 넘는 AAB 를 AppleScript 한 줄로 업로드 못 해서** base64 청크 28개로 쪼갰다.
> 자정 무렵, 텔레그램 입력중 데몬이 오펀으로 쌓이는 구조적 결함을 잡고 Stop 훅 알림 방향을 재설계했다.

이 날은 **메모요 양대 앱스토어 출시를 실제로 끝내는 한 호흡** 이었다. 아침까지 준비해 둔 메타데이터와 ipa 위에, 저녁부터 iOS 제출·Android 서명 설정·Play Console 전 과정 자동화·스토어 자산 생성·베타 모집 글까지 동시에 쳐냈다. 덤으로 운영 환경 쪽 크고 작은 결함 세 개를 같이 정리했다. 이 버전은 그 압축적인 밤을 다시 나눈 소급판이다.

## 1. iOS: Transporter 로 올리고 바로 심사 제출

절차는 깔끔히 흘러갔다.

- Transporter 로 `메모요.ipa` 업로드 → App Store Connect 에 빌드 노출 확인
- 메타데이터는 사전에 `docs/store-assets` 에 정리해 둔 **이름·부제·설명문·키워드·프로모션 텍스트·개인정보 처리방침 링크** 를 그대로 붙여넣기
- 실기기 스크린샷을 9:16 비율로 정리해 업로드
- 연령 등급·앱 프라이버시·데이터 안전 항목 입력
- 심사 제출 → 상태 **`WAITING_FOR_REVIEW`** 전환 확인

iOS 내 작업은 여기서 일단락. 결과는 24~48시간 안에 떨어질 예정.

## 2. Android: 서명이 없으면 AAB 가 없다

Android 는 상황이 달랐다. **릴리즈 서명이 없어서 AAB 를 못 만드는 상태**. 처음부터 끝까지 세팅해야 했다.

### 2.1 키스토어 생성

`keytool` 로 `memoyo-upload-keystore.jks` 를 새로 만들었다.

```
Algorithm  : RSA 2048-bit
Validity   : 10,000 days
Alias      : upload
```

같은 폴더에 `key.properties` 를 박아 `storePassword`, `keyPassword`, `keyAlias`, `storeFile` 네 값을 심었다.

### 2.2 Gradle 서명 주입

`app/build.gradle.kts` 상단에 Properties + FileInputStream import. `rootProject.file` 로 keystoreProperties 를 읽어오는 블록 추가. `android` 블록 안에 `signingConfigs { create("release") { ... } }` 를 새로 넣고 네 값을 `keystoreProperties` 에서 꺼내 매핑.

`buildTypes.release` 에서는 기존 debug 서명을 release 로 교체하고 **`minifyEnabled = true`, `shrinkResources = true`** 로 올려 프로가드까지 엮었다.

```bash
flutter build appbundle --release
```

서명된 AAB 정상 생성 확인.

## 3. Play Console 전 과정을 JS 인젝션으로 자동화

Chrome + AppleScript + 브라우저 JS 인젝션으로 전 과정을 자동화했다.

- 계정 대시보드에서 앱 생성 → 메모요 이름·한국어 기본 언어·무료·앱 선언 라디오/체크박스를 프로그램적으로 클릭
- 대시보드 작업 리스트의 **11가지 선언** 차례대로 처리: 스토어 등록정보, 앱 콘텐츠, 연령 등급, 대상 사용자 층, 데이터 안전성, 광고 유무, 뉴스앱 여부, 코비드19 앱 여부, 정부앱 여부, 금융앱 여부
- 대부분 "아니요" 로만 찍으면 됐지만 **저장 버튼이 비활성으로 꼬이는 구간** 이 몇 번 있어 저장을 먼저 누른 뒤 다음으로 넘어가는 방식으로 우회
- 콘텐츠 등급 IARC 설문 9문항 모두 "아니요" → **전체 이용가 등급** 획득

## 4. 40MB AAB 를 청크 28개로 쪼개 업로드

AAB 업로드에서 예상 못 한 벽. **파일 크기 40MB+ 가 AppleScript 단일 호출에 안 들어감**.

우회:

1. 파이썬으로 AAB 를 base64 인코딩
2. **2MB 청크 28개** 로 쪼갬
3. Chrome 의 `window` 객체에 `window.aab_0` 부터 `window.aab_27` 까지 순서대로 얹음
4. 조립 스크립트로 Blob 을 만들고 `DataTransfer` 에 `File` 오브젝트를 밀어 넣어 `<input type="file">` 에 프로그램적으로 할당

이 과정을 반복해 1.0.0 릴리즈 AAB 가 내부 테스트 트랙에 정상 업로드. 저장·검토·출시까지 눌러 **내부 테스트 상태가 "심사 대기"** 로 전환.

> **배운 것** — 브라우저 API 한계(문자열 길이·DataTransfer 구조) 와 AppleScript 인자 한계가 만나면 **binary 를 base64 로 쪼개 window 스코프에 나눠 올리는 게 가장 단순한 우회다**. 라이브러리도 드라이버도 필요 없다.

## 5. 개인 계정은 프로덕션 대신 클로즈드 테스트부터

Google Play 의 2023-11 정책 변경 이후 **개인 개발자 계정** 은 프로덕션을 바로 열 수 없다. **20명 테스터로 14일간 클로즈드 테스트** 를 먼저 돌려야 한다. 내부 테스트는 즉시 가능해 일단 그쪽으로 릴리즈 걸어 둔 상태.

남은 건 테스터 모집. 한국어 커뮤니티용 베타 모집글을 완성형으로 쓰고 오픈카카오톡방에 바로 올렸다. 앱 소개 + 5단계 참여 안내 + 내부 테스트 옵트인 링크 + 감사 DM + iOS 출시 알림 제공 + 연락 이메일. 후속 모집 대상 후보: 클리앙, 뽐뿌, 디시, 네이버 카페, 트위터.

## 7. 오펀 typing 데몬을 잡다

운영 환경 쪽 결함 하나. **텔레그램 입력중 지시기가 세션이 끝나도 계속 살아있는** 오펀 데몬 문제.

원인 구조:
- `telegram-typing-start.sh` 훅이 idempotent 기준이 약해 **이전 세션 데몬이 남아 있으면 신규 세션 시작 시 또 하나가 붙음**
- Stop 훅은 PID 파일에 기록된 **하나만 죽임** → 오펀이 쌓임

조치: 두 스크립트를 **pkill 패턴 매칭 기반** 으로 재작성해 세션 시작·종료 시 `telegram-typing-daemon.sh` 프로세스를 전부 죽이도록 바꿈. (이 광범위 pkill 방식이 4/20 밤 "세션 격리 버전" 으로 한 번 더 뒤집히는 원형이다.)

## 8. Stop 훅 알림 방향 재설계

더 미묘한 결함. 기존 `telegram-stop-ping.sh` 는 **transcript 어디에라도 `plugin:telegram:telegram` 태그가 있으면 무조건 푸시를 스킵**. 텔레그램 플러그인이 붙은 세션이면 **터미널에서 입력한 답변조차** 푸시가 안 갔다.

사용자는 이 경우에도 "답변 왔습니다" 알림이 필요했다. 방향 재설계:

- **마지막 user 메시지의 content 만** `jq` 로 추출
- 그 안에만 텔레그램 origin 마커가 있는지 체크
- 메시지가 array 형태의 `tool_result` 가 아닌 **string 타입 user 입력만** 필터링
- `origin.server` 필드의 `plugin:telegram:telegram` 문자열을 grep

이제 텔레그램 입력에 대한 응답은 (봇 reply 푸시와 중복되는) 훅 알림 스킵, 터미널 입력에 대한 응답은 정상적으로 푸시가 울림. 메모리 파일도 이 규칙으로 덮어씀.

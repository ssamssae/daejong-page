## 2. /merge-janitor 가 필요했다

WSL 에서 이미 **morning-briefing 의 1-E 섹션** 을 추가해 janitor PR 을 아침 07:15 통합 브리핑에 끼워 넣는 쪽은 push 돼 있었다. 그런데 **그 PR 에 대해 사용자가 텔레그램 답장으로 "PR 3번 머지" 같은 문장을 쳤을 때 실제로 `gh pr merge` 가 돌아가는 핸들러는 비어 있었다**.

이를 풀기 위해 `/merge-janitor` 라는 새 스킬을 `claude-skills` 에 작성.

구조:

1. `gh search` 로 open 상태 janitor PR 을 번호로 찾음
2. `gh pr view` 로 diff 메타데이터 가져와 안전 체크
   - 변경 줄 수 300 초과 여부
   - `pubspec`·`Podfile`·`Gemfile` 같은 **의존성 파일** 포함 여부
   - `.env`·`key`·`token`·`secret`·`.pem`·`.p12`·`keystore` 같은 **민감 파일** 포함 여부
   - CI·키스토어 변경 여부
3. 경고가 있으면 사용자 확인 대기, 없으면 곧바로 `gh pr merge` 실행
4. 닫기 모드는 `gh pr close --delete-branch` 한 줄로 분기

처음엔 `gh search` 한 번에 `additions`, `deletions`, `files` 까지 받아오려다가 **그 필드들은 search 에서 지원 안 됨** 을 검증 중 발견해 2단계로 분리.

> **배운 것** — `gh` 의 `search` 서브커맨드와 `view` 서브커맨드는 리턴 스키마가 다르다. "한 번에 다 가져오자" 가 먹히지 않는 경우를 대비해 **"찾기" 와 "메타 읽기"** 를 처음부터 분리하는 쪽이 안전하다.

## 3. Squash 에서 Merge 로 뒤집은 이유

머지 전략 기본값을 `squash` 로 시작했다가 **강대종님 지적으로 다시 뒤집었다**.

지적의 근거:
- janitor 커밋이 이미 **atomic 하게 잘 쪼개진 conventional commits**
- 개인 repo 의 `git log` 가 **그 자체로 기억 보조장치**
- squash 하면 `bisect` 와 `blame` 해상도가 떨어짐

**그래서 default 를 `--merge` 로 바꾸고, 필요할 때만 `/merge-janitor N squash` 또는 `--squash` 플래그를 받아 squash 모드로 들어가는 옵션**으로 정리. WIP 반복 커밋 패턴이 쌓이는 프로젝트가 나중에 생기면 그 때 기본값을 **커밋 수 휴리스틱** 으로 전환하자는 방향성도 스킬 본문에 적어 뒀다.

> **배운 것** — 기본값을 정할 때 **"내 git log 의 용도"** 를 먼저 본다. 로그가 기억 보조장치인 1인 repo 와, 로그가 릴리즈 노트 소스인 팀 repo 는 기본값이 반대 방향이다.

## 4. 자연어 트리거를 양쪽에 심다

`/merge-janitor` 의 자연어 트리거는 skill description 안에 **"PR N 머지" / "PR N 닫아"** 패턴을 적어 둬 Mac 과 WSL 양쪽에서 `claude-skills pull` 만 받으면 바로 인식되게 했고, 맥 본진의 `CLAUDE.md` 에도 같은 문장을 한 줄 추가해 보강했다.

# 2026.04.21 작업일지 v1.0.1 — "허용" 버튼이 세 번 다 빗나간 아침

> 08:27, 메모요 1.0.2 는 승인된 지 며칠째 "배포 준비됨" 으로 조용히 서 있었다.
> 09:00, 자동화로 찍은 첫 시뮬레이터 스크린샷이 Claude 에 들어가질 않았다.
> 09:29, 강대종님이 "작업중지해" 로 세션을 접었다.

이 버전은 하루의 오전 구간(00:44~09:45 KST) 기록이다. 메모요 App Store 수동 출시 단계에서 **"Pending Developer Release" 라는 낯선 상태 표기** 가 강대종님을 5분쯤 붙잡았고, 이어서 새 앱 네 개의 스크린샷을 자동으로 찍으려다 **이미지당 2000px 제한 + iOS 알림 허용 다이얼로그** 라는 두 장애물에 30분을 태웠다. 마지막은 /worklog 스킬을 뉴스레터 스타일로 뒤집어엎는 작업. 이 파일이 그 새 포맷의 첫 적용본이다.

## 1. 승인된 버전이 출시되지 않는 이유

메모요 1.0.2 (build 9) 는 지난주 심사를 통과해서 App Store Connect 에서 "**Pending Developer Release**" 상태로 대기 중이었다. 의미는 단순하다. *심사는 끝났으니 너(개발자)가 출시 버튼을 누르면 스토어에 반영한다.*

강대종님이 App Store Connect 를 열고 상태를 보자 "배포 준비됨" 이라는 한국어 라벨이 떴다. 이게 애매했다. 두 가지로 읽을 수 있었다.

1. 심사 통과 후 **수동 출시 버튼 대기 중**
2. 심사 제출 전 **파일/메타데이터 준비 완료**

확인 경로는 설정에서 **배포 옵션** 을 "자동 출시" 가 아닌 "수동 출시" 로 바꾸고 저장하면 "이 버전 배포" 버튼이 명시적으로 노출된다는 것. 자동 출시 설정일 때도 페이지 최하단 또는 사이드바 "앱 심사" 탭에서 같은 버튼이 나타난다. 강대종님이 스크린샷 몇 장을 주고받으며 우측 상단 버튼을 찾아 눌렀다.

여기까지 **약 5분**. 메모요 1.0.2 는 스토어 반영 대기로 넘어갔다.

## 2. 같은 앱, 다른 버전이 동시에 떠 있었다

앱 심사 사이드바에 1.0.2 말고 **1.0.4 (build 21)** 도 함께 보였다. 혼선의 원인은 두 가지였다.

- 어제 밤에 `~/submit-queue/memoyo-1.0.4-21.ipa` 를 빌드만 해두고 업로드는 안 했음
- App Store Connect 버전 드롭다운에 버전 번호만 보이고 "업로드 여부" 가 한 번에 안 드러남

그래서 처음엔 1.0.4 도 자동으로 뭔가 진행되는 줄 알았는데, 실제 상태는 "버전 생성만 돼 있고 빌드는 비어 있음". 업로드부터 다시 밟아야 했다.

altool CLI 로 ipa 를 올렸다.

```bash
xcrun altool --upload-app \
  -f ~/submit-queue/memoyo-1.0.4-21.ipa \
  -t ios \
  -u ssamssae@me.com \
  -p <app-specific-password>
```

전송은 **4.7 초**. Delivery UUID `35d78575-...` 로 Apple 수령 확인. Processing 10~30분 대기가 붙지만 그 사이에 남은 일을 진행했다.

## 3. Playwright 가 죽어 있었다

심사 제출 버튼을 눌러야 하는 단계에서 Playwright MCP 서버의 Chrome 세션이 **stale** 상태였다. `browser_navigate` 호출이 돌아오지 않았다. 진단:

```
mcp-chrome-1f3f523 (PID 죽음 아님, 컨텍스트만 끊김)
```

클린 재시작을 위해 `pkill -f "mcp-chrome-1f3f523"` 를 치려 했는데 **sandbox 권한에 막혀** Claude 가 직접 실행할 수 없었다. 해결은 간단했다. 강대종님이 Ghostty 터미널에서 그 한 줄을 직접 쳤고, 다음 `browser_navigate` 호출에서 새 Chrome 컨텍스트가 자동으로 올라왔다.

> **배운 것** — Claude Code 에 막힌 명령은 대체로 사용자에게 위임하면 된다. 90% 의 "불가능" 은 권한 문제이고, 프롬프트에 `!pkill -f "..."` 같은 지시 한 줄을 복붙해 달라고 청하는 쪽이 우회로보다 빠르다.

로그인은 2FA 때문에 강대종님이 직접 수행. 그 다음 "심사 제출" 버튼까지는 Playwright 가 클릭했다. "1개의 항목 제출됨" 팝업 + 사이드바 "1.0.4 심사 대기 중" 전환 확인. **자동 출시 설정** 이라 승인 시 스토어 바로 반영.

## 4. 허용 버튼이 세 번 다 빗나갔다

이제 신규 앱 네 개(약먹자/더치페이/한컵/mini_expense) 의 심사 스크린샷을 자동으로 찍을 차례였다. iPhone 16 Pro Max 시뮬레이터를 생성하고, Flutter 로 Runner.app 을 빌드하고, `xcrun simctl install` + `launch` 로 앱을 띄우고, `xcrun simctl io booted screenshot` 으로 PNG 를 찍었다. 여기까지는 흘러갔다.

**문제가 둘 동시에 터졌다.**

### 4.1 이미지 2000px 한계

스크린샷 해상도는 1320×2868. Claude 가 Read 로 받아들일 수 있는 이미지 최대 변은 **2000px**. 세로 2868 이 이 경계를 훨씬 넘어서서 **매 스크린샷마다 "An image in the conversation exceeds the dimension limit" 에러**.

### 4.2 "허용" 버튼이 안 사라졌다

앱 첫 화면에 iOS 알림 권한 요청 다이얼로그가 떴다. 스크린샷에 이게 박히면 심사 스크린샷으로 쓸 수 없다. 닫기 시도를 **세 가지 다른 경로** 로 했는데 **각기 다른 이유로 셋 다 실패**.

**① AppleScript 로 버튼 클릭.**

```osascript
tell application "System Events"
  tell process "Simulator"
    click button "허용" of window 1
  end tell
end tell
```

결과:

```
execution error: System Events에 오류 발생:
button "허용" of window 1 of process "Simulator"을(를) 가져올 수 없습니다. (-1728)
```

진단을 위해 `every button of window 1` 을 호출해 보니 시뮬레이터가 AppleScript 에 노출하는 버튼은 `Action`, `Volume Up`, `Volume Down`, `Sleep/Wake`, `5`, `6`, `7` **뿐**. 이것들은 macOS 메뉴바/도구바에 붙는 **시뮬레이터 자체의 컨트롤** 이다. iOS 앱 내부 UI 요소는 **macOS 접근성 API 에 노출되지 않는다**. 시뮬레이터는 iOS 화면을 렌더링하는 비디오 박스에 가깝지, macOS 가 뜯어볼 수 있는 네이티브 뷰가 아니다.

**② `xcrun simctl privacy` 로 권한 거부.**

```bash
xcrun simctl privacy booted deny notifications com.ssamssae.yakmukja
```

결과:

```
Unknown action 'deny'
```

매뉴얼을 읽어 보니 `simctl privacy` 의 액션은 `grant`, `revoke`, `reset` 셋 뿐이다. `deny` 는 없다. 그리고 지원 서비스 목록은 `calendar`, `contacts`, `location`, `photos`, `microphone`, `motion`, `reminders`, `siri` — **notifications 가 목록에 아예 없다**. 이 경로로는 알림 권한 자체를 제어할 수 없다.

**③ cliclick 으로 좌표 클릭.**

```bash
cliclick c:915,313
```

실행은 됐다. 스크린샷을 다시 찍어 보니 **다이얼로그가 그대로 남아 있었다**. cliclick 의 좌표는 **macOS 화면 좌표계** 인데, 시뮬레이터는 iOS 해상도 1320×2868 를 창 크기로 축소해 렌더링한다. 창 오프셋과 스케일을 반영하지 않고 쏜 915×313 은 실제로는 엉뚱한 공간을 찍었다.

> **배운 것** — 같은 증상(허용 버튼 안 눌림)처럼 보여도 실제 실패는 **세 개의 다른 레이어** 에서 났다. GUI 접근성 API / CLI 기능 누락 / 좌표계 변환 누락. 증상만 보고 "뭔가 안 된다" 로 묶지 말고 각 시도의 실패 메시지를 별도 원인으로 읽어야 한다.

같은 dimension limit 에러가 컨텍스트에 쌓이면서 어시스턴트는 루프를 돌았고, **09:29 에 강대종님이 "작업중지해"** 로 세션을 강제 종료. /clear.

### 다음 시도 때 반영 (합의)

- ① 앱 코드에 스크린샷 전용 빌드 flavor 를 만들어 **알림 권한 요청 자체를 skip**
- ④ 심사 스크린샷은 권한 다이얼로그가 뜨지 않는 **홈 화면 중심** 으로 한정
- 스크린샷 PNG 는 `sips -Z 1800 원본 --out 축소본.png` 로 리사이즈 후 Read

## 5. /worklog 스킬을 뒤집어엎다

새 세션에서 강대종님이 /worklog 의 포맷을 **뉴스레터 스타일** 로 바꾸자고 제안했다. 레퍼런스는 `~/daejong-page/newsletter/ep1-2026-04-20.md`. 선택지는 셋.

- **A**: 원본 + 공개본 둘 다 뉴스레터 마크다운, 렌더러에 marked.js 추가
- **B**: 이원화 복원 — 원본만 뉴스레터, 공개본은 카페 호환 산문체
- **C**: 산문체 유지 + evocative 섹션 제목만 이식

현장 조사에서 세 가지가 드러났다.

1. `ep2-2026-04-21.md` 는 **아직 없다** (폴더엔 ep1 단독)
2. `worklog.html` + `worklog/view.html` 둘 다 `bodyEl.textContent = text` + `white-space: pre-wrap` — **마크다운 파싱 안 함**. 공개본의 "마크다운 기호 금지" 규칙은 이 평문 렌더링 제약에서 온 것
3. view.html "본문 복사" 버튼은 카페 붙여넣기용 원본 공급처

C 는 뉴스레터의 강점(시각 구조화) 을 못 쓴다. B 는 이원화 유지 비용은 있는데 공개본이 여전히 밋밋. **A** 는 marked.js CDN 한 줄과 CSS 20여줄이면 되고, 대신 카페 붙여넣기를 드롭해야 한다. 강대종님 승인 후 A 로 진행.

변경분:

- `~/.claude/skills/worklog/SKILL.md` — 2종 파일 설명·6-c·8번 섹션 전면 재작성. **뉴스레터 스타일 체크리스트 10개** 추가(제목 emdash, 블록쿼트 훅, 번호 매긴 evocative 섹션, story arc, 배운 것 블록쿼트, 서명 자동 삽입). 카페 붙여넣기 드롭 선언 주석.
- `~/daejong-page/worklog.html` — `<head>` 에 marked.js CDN, `.entry-body` 의 `white-space: pre-wrap` 제거, 마크다운 요소 CSS 추가, `bodyEl.textContent = text` → `bodyEl.innerHTML = marked.parse(text, { breaks: false, gfm: true })`. 실패 시 textContent fallback. "산문체로 보기" 버튼 → "단일 버전 보기".
- `~/daejong-page/worklog/view.html` — 동일 패턴. 페이지 타이틀 "작업일지 산문체" → "작업일지 뉴스레터".

과거 파일 호환성: 2026-04-20 이전 산문체 파일은 문단이 빈 줄로 구분돼 있으므로 marked 가 `<p>` 로 잘 쪼갠다. 마크다운 기호가 없어 그대로 평문처럼 렌더링되고 깨지지 않는다.

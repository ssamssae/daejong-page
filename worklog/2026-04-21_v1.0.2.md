## 2. 실측이 뒤집은 가설

나는 "매시 실행되는 `claude -p /review-status-check` 가 매번 풀 Claude 세션을 띄우니까 토큰 꽤 쓸 것" 이라고 가정하고 답을 쓰려다 멈추고 실제 로그를 열어봤다:

```
/tmp/claude-review-status-cron-wrapper.log
```

- **PROCEED**: 1회 (4/18 17:12)
- **SKIP**: 37회

다 스킵되고 있었다. 이유는 wrapper 스크립트가 `claude --channels` 인터랙티브 세션이 살아있으면 "텔레그램 봇 getUpdates 409 Conflict 방지" 로 자동 skip 하게 돼 있었고, 대종님은 tmux 에 `claude --channels` 를 상시 띄워놓는 구조라 100% 걸렸다.

> **배운 것** — "토큰 낭비되는 거 아니야?" 질문에 추측으로 답하면 틀린다. `/tmp/` 의 실제 실행 로그를 열어 PROCEED:SKIP 비율부터 측정하는 게 먼저.

실 토큰 사용량은 ≈ 0. 하지만 대종님이 물은 건 단순 비용이 아니라 "이게 살아있어야 하나" 였다. 구조적 답변은 필요했다. 선택지 4개 제시:

1. 현상 유지 (어차피 SKIP)
2. Pure bash 재작성 (Gmail IMAP + App Password) — 진짜 0 토큰
3. launchd unload 후 수동 호출만
4. 완전히 다른 접근

제일 경제적 답은 **3번**. 대종님이 "D" 로 채택.

## 3. D — 삭제가 아니라 격리

`launchctl unload` 를 때렸다. I/O 에러가 로그에 찍혔지만 효과는 있었다. plist 파일을 `~/Library/LaunchAgents/_disabled/` 아카이브 디렉토리로 이동하는 패턴을 쓴다. 완전 삭제가 아니라 **격리**. 되돌리려면 한 줄:

```bash
mv ~/Library/LaunchAgents/_disabled/com.claude.review-status-check.plist ~/Library/LaunchAgents/
launchctl load ~/Library/LaunchAgents/com.claude.review-status-check.plist
```

`/review-status-check` 스킬 자체는 남겨뒀다. 수동 호출은 언제든 가능. 자동만 끔.

## 5. 캐스케이드

**1. 3개 스킬 삭제**. `claude-skills` 저장소에서 `rm -rf` 후 push. 대종님 홈페이지 `skills.html` 에서도 3개 카드 제거. 이 대화 시스템 프롬프트에서 즉시 3개가 사라진 걸로 확인.

**A. launchd 통합**. 새 스킬 `/evening-wrap` 작성 — `done` 이 체크리스트 만들고 `todo-reminder` 가 내일 우선순위 리마인드까지 **한 세션** 안에 처리. 22:30 에 신설 plist 로 실행. 기존 두 plist 격리. 매일 Claude 세션 하나 통째로 절약.

작업 중간에 예상 못한 발견:

```bash
$ launchctl list | grep com.claude
-  0  com.claude.evening-wrap
```

`morning-briefing`, `daily-sync-and-learn` 이 list 에서 빠져있었다. plist 파일은 있는데 launchd 에 등록이 안 된 상태. 언제부터 그랬는지는 불명. 과거의 `unload` 시도가 그 옆 잡까지 튕겼을 가능성. `launchctl load` 로 둘 다 복구.

> **배운 것** — launchd 작업을 손볼 때는 **손대기 전에** `launchctl list | grep <prefix>` 로 현재 등록 상태 스냅샷을 찍고, 손댄 후 다시 확인. 그렇게 안 하면 옆 잡이 조용히 빠져있는 걸 몇 주씩 모르고 지나간다.

**B. MEMORY.md 중복 통합**. 두 쌍이 주제 겹침 — 드림팀(용어 + 워크플로우 2파일), 텔레그램 stop 훅(동작 규칙 + race retry 2파일). 각각 하나로 합침. 31파일→29, 30줄→28.

**C. night-runner 메모 축약**. 4/19 작성 당시의 "내일 07:15 첫 검증" 상태가 그대로 박혀 있었음. 이미 검증 지난 시점이라 운영 규칙만 남겨 압축. 16줄→10.

## 7. 막힌 자리, 허락의 형태

WSL 세션이 1단계에서 막혔다:

> 🛑 1단계 진행 중 브릿지에 막힘 — 확인 부탁드려요.
> arun/, night-runner/, trend/ 가 로컬 untracked 로 존재해서 pull 이 충돌. diff 확인해보니 origin/main 과 완전 동일. rm 후 pull 하면 손실 0.

Claude Code 의 안전장치는 기기 역할(Mac 본진 / WSL 전위)과 무관하다. 세션 시작 전부터 존재하던 파일을 `rm` 하는 것 = 사용자 작업 삭제 위험으로 기본 거부. WSL 이 "전위" 라는 말은 빌드 부하·실패 허용 범위 얘기지, 삭제 권한 완화가 아니다.

대종님이 "진행 OK" 한 줄로 풀렸다.

> **배운 것** — 멀티 디바이스 자동화에서 "어느 기기가 공격적이냐" 는 역할 설계고, "어느 경로가 비가역이냐" 는 권한 설계다. 섞으면 안 된다.

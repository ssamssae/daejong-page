## 3. Xcode 가 라이선스에 동의하라고 했다

Android appbundle 을 빌드하는데 Xcode 가 튀어나왔다.

```
You have not agreed to the Xcode license agreements.
Please run 'sudo xcodebuild -license' from within a Terminal window.
```

Flutter 는 Android 빌드에서도 `xcrun` 체크를 돌린다 — 엔진 체인 어딘가에 엮여 있다. `sudo xcodebuild -license accept` 는 interactive sudo 비밀번호가 필요해서 Claude 세션에선 막혔다. 우회는 환경변수 한 줄이었다.

```bash
DEVELOPER_DIR=/Library/Developer/CommandLineTools \
  flutter build appbundle --release
```

Flutter 가 Xcode.app 대신 CommandLineTools 쪽 developer dir 을 보면서 라이선스 체크를 건너뛴다. 11초 뒤 `app-release.aab 46.3MB` 가 떨어졌다.

> **배운 것** — Flutter 의 플랫폼 간 오염은 생각보다 깊다. Android 빌드가 Xcode 라이선스에 막히는 일이 일어난다. `DEVELOPER_DIR` 환경변수로 CommandLineTools 를 주입하면 sudo 없이 풀린다.

## 4. 네 번의 업로드, 네 번의 다른 에러 메시지

`~/.claude/automations/scripts/play-upload.py` 한 방이면 끝나야 했는데, 네 번 튕겼다. 매번 다른 이유였다.

```
# 시도 1: vc=3, --track beta
bundles.upload → 403 "Version code 3 has already been used"
```

vc=3 은 과거 어느 시점에 이미 Play Console 에 등록돼 있었다. pubspec 을 `1.0.0+4` 로 올리고 재빌드.

```
# 시도 2: vc=4, --track beta
bundles.upload ✅
tracks.update → 400 "Precondition check failed"
```

이건 `lesson: android-beta-track-needs-testers.md` 에 정확히 적혀있던 증상이었다. Closed Testing 트랙(alpha/beta) 은 testers 그룹이 enrolled 돼있어야 release 생성을 허용한다. 스크립트의 `status=draft` auto-retry 는 다른 에러 케이스용이라 이 타입은 못 풀었다.

강대종님이 Play Console 웹에서 alpha 트랙에 `memoyo-beta-testers@googlegroups.com` 그룹을 붙여 5분 만에 해결. 스크린샷을 받고 재시도.

```
# 시도 3: vc=5, --track alpha
bundles.upload ✅
tracks.update (status=completed) → 400 "Only releases with status draft may be created on draft app"
auto-retry (status=draft) → 새 edit 열어 bundles.upload → 또 실패 (vc=5 이미 소비)
```

"draft app" 제약이 튀어나왔다. 포모도로는 아직 publish 된 적 없는 "앱 초안" 상태라 모든 새 릴리즈가 `status=draft` 여야 한다. 스크립트는 이 에러도 알고 있었고 `status=draft` 로 재시도까지 한다 — 근데 이번엔 재시도가 새 edit 을 열면서 같은 aab 를 다시 업로드하려 해서 "versionCode already used" 에 걸려 자멸했다.

```
# 시도 4: vc=6, --track alpha, --status draft (처음부터)
bundles.upload ✅
tracks.update ✅
edits.commit → 400 "Only releases with status draft may be created on draft app"
```

처음부터 `--status draft` 를 줬는데도 **commit** 단계에서 같은 에러가 나왔다. 여기서부터는 알려진 lesson 으로 풀리지 않는 구간이었다.

## 5. vc=3 completed 가 내부 트랙에 걸려 있었다

스크립트를 덮고 수동 파이썬으로 들어갔다. 모든 트랙을 순회하며 현재 상태를 찍어봤다.

```
internal: releases=1
  - name=3 status=completed vc=['3']    ← 이것
alpha:    releases=0
beta:     releases=0
production: releases=0
```

숨어있던 건 **internal 트랙의 vc=3 status=completed** 였다. 시도 1 이 bundles.upload 에서 튕길 때, 그 이전 어느 세션에서 internal 트랙까지 타고 들어가 status=completed 로 commit 된 릴리즈가 남아 있었다. 지금의 edit 은 alpha 트랙만 건드렸지만, Play Developer API 의 `edits.commit` 은 edit 안의 **모든 트랙 전체 스냅샷을 validate** 한다. "draft app 은 모든 릴리즈가 draft 여야" 규칙이, 우리가 건드리지도 않은 internal 트랙의 완료 릴리즈를 보고 거절을 내렸던 것이다.

에러 메시지 "Only releases with status draft **may be created**" 는 오해를 유발했다. 실제로는 "may **exist**" 에 가까웠다. 우리가 지금 생성하는 릴리즈만 검사하는 게 아니라, 커밋 후의 전체 앱 상태를 검사한다.

> **배운 것** — Google Play Developer API 의 에러 메시지는 가끔 주어를 속인다. "may be created" 가 "may exist" 를 뜻할 때가 있다. draft app 에 non-draft 릴리즈가 한 군데라도 남아있으면 다른 모든 commit 을 막는다.

해결은 단순했다. 같은 edit 안에서 internal 트랙의 vc=3 릴리즈도 draft 로 demote 한 뒤 commit.

```python
edit = service.edits().insert(packageName=pkg, body={}).execute()
bundles.upload(vc=6)                                                  # OK
tracks.update(internal, releases=[{name:'3', vc:['3'], status:'draft'}])   # demote
tracks.update(alpha,    releases=[{name:'6', vc:['6'], status:'draft', notes:...}])
edits.commit(eid)                                                      # ✅ OK
```

edit_id `04724584959853325210` 가 커밋됐다. 06:30 KST. alpha 트랙에 vc=6 draft 릴리즈가 들어앉았고, internal 의 vc=3 도 draft 로 내려갔다.

## 5. 두 봇은 텔레그램으로 서로 못 본다

> 18:34, "로미오와 줄리엣같구만 서로 말걸게하기가 이렇게 어려울 줄이야 ㅋㅋ"

오후에 시도한 peer-bot direct POST 가 텔레그램 정책에 걸렸다. 봇 A 가 봇 B 의 채팅에 메시지를 직접 송신할 수 없고, 그룹에 둘 다 넣어도 봇끼리 가시성은 차단된다 (privacy mode off 해도 동일). 강대종님 user_id 를 매개로 한쪽 봇이 메시지를 쏘면, 다른 봇 측 챗에는 그 메시지가 안 보인다.

이걸 알게 된 후 `/handoff` 스킬을 정직하게 다시 썼다 (`b1e5608`):

> 자동전달은 못 한다. 이 스킬은 "챗 전환 1회 제거" UX 개선만 한다. 강대종님이 챗 전환 없이 한 챗에서 long-press → copy → paste → send 로 끝낼 수 있게.

그러니까 텔레그램 채널은 죽은 카드였다. 다른 채널이 필요했다.

## 6. METHOD A — 0.5초의 간격

> 18:36, "안되는데 방법을 찾아서 어떻게된 성공시키는 이 과정 이걸 즐기는게 재밌어"

저녁 8시 46분, SSH + tmux send-keys 로 다른 기기 Claude Code 프롬프트에 인젝션하는 첫 시도 (`fa68357`). 동작 자체는 됐다. 텍스트가 도착했다. 그런데 Enter 가 submit 을 안 시켰다.

```bash
ssh user@mac /opt/homebrew/bin/tmux send-keys -t claude-main 'text' Enter
```

이걸 한 burst 로 보내면 텍스트만 들어가고 줄바꿈이 추가될 뿐이었다. `Enter Enter` (`387d781`) 도 안 됐다. `S-Enter` (Shift+Enter) 도 똑같이 흡수됐다.

원인은 bracketed paste 였다. tmux 가 빠른 burst 를 통째로 paste 마커 (`\e[200~ ... \e[201~`) 안에 감싸 보내면, 마커 안의 모든 Enter/S-Enter 는 paste 콘텐츠로 처리된다 — 즉 newline 으로 들어간다. Claude Code 의 multi-line input 박스가 그걸 그대로 받아들이고, submit 은 마커 밖에서 와야 하는 진짜 keystroke 만 인식한다.

해결 (`a75cef1`) — text 와 Enter 를 sleep 으로 분리:

```bash
ssh host "tmux send-keys -t SESSION 'text'; sleep 0.5; tmux send-keys -t SESSION Enter"
```

0.5초 동안 paste 모드가 종료된다. 다음 Enter 는 진짜 keystroke 로 도착한다. submit 발화. 21:37 KST METHOD A PASS. 그 직후 양방향 verify directive (`655b60b`), Mac PATH 함정 (`/opt/homebrew/bin/tmux` 절대경로 — 비인터랙티브 SSH 셸은 zshrc 안 로드), 양방향 대칭 셋업 (`f4f9cf7`).

22:50 KST, **WSL → Mac PING 도착, PONG 회신**. 텔레그램 손가락 없이.

> **배운 것** — bracketed paste 는 "사용자가 붙여넣기한 한 덩어리" 를 식별하기 위한 메커니즘인데, 자동화 입장에선 keystroke 와 paste 의 경계를 흐리는 함정. tmux 자동 인젝션에서 "text 와 control key 사이엔 한 박자 쉬어야 한다" 가 일반 패턴.

# 2026.04.29 작업일지 v1.0.0 — Mac mini 가 책상에서 빌드 머신으로 진급한 날

> 09:53, Mac mini 야간 빌드 보고서엔 PASS:0 FAIL:4 가 찍혀 있었다.
> 11:53, 같은 4앱이 PASS:4 FAIL:0 으로 돌아왔다.
> 21:14, 같은 머신이 hanjul.ipa 19MB 까지 뽑아냈다.

오늘은 Mac mini 가 단순히 24/7 켜져있는 fallback 노드에서, **Android release 빌드의 SoT + iOS ipa 빌드 머신**으로 진급한 날이다. 어제까지의 sign NPE 미스터리가 풀렸고, /handoff 184줄 스킬은 wrapper 1개로 줄었으며, /night-runner v1 안전모드가 03:00 KST launchd 에 실리며 오늘 밤부터 매일 자동으로 깨어난다. 인프라 정비 한 사이클이 끝난 진짜 큰 날.

## 1. sign NPE 의 정체는 단순했다 — keystore 가 없었다

어제(4-28) Mac mini 첫 야간 빌드 v1.0 의 결과는 PASS:0 FAIL:4. 4앱 모두 sign NPE 로 죽었다. 어제 원인 분석으로 "release keystore 가 Mac mini 에 없으니까 서명 단계가 NullPointerException" 이라는 가설이 섰는데, 오늘 그걸 확정하고 처방까지 박았다.

처방은 단순:

```
Mac 본진  ~/apps/<app>/android/{key.properties, *-upload-keystore.jks}
   ↓ rsync (Tailscale internal SSH only)
Mac mini  ~/apps/<app>/android/  (chmod 600)
```

4쌍 8파일을 옮긴 뒤 hanjul 단일 검증 빌드 — PASS(32.3s). 이어서 launchctl 로 night-build 일괄 → Gradle daemon + dep cache 효과로 1분 안에 4앱 모두 끝났다.

| 앱 | 결과 | 시간 | aab |
|---|---|---|---|
| hanjul | ✅ | 4s | 42.6MB |
| hankeup | ✅ | 6s | 44.1MB |
| mini_expense | ✅ | 7s | 45.7MB |
| pomodoro | ✅ | 7s | 46.3MB |

**PASS: 4 / FAIL: 0.**

이참에 정책도 명문화했다 — **Mac mini = Android release 빌드 SoT, WSL = debug only.** WSL 에선 `flutter build appbundle --release` 실행 금지. keystore 없이 release 시도하면 sign NPE 가 다시 떠서 가짜 PASS 가 나거나 진짜 빌드 실패를 만든다. 평시 keystore 단일 위치 = Mac mini, 본진 archive(`~/keystores-archive/`) 는 안전망일 뿐.

> **배운 것** — sign NPE 같은 추상적 에러는 "왜 NPE 인가" 보다 "**어디에 있어야 할 것이 없는가**" 부터 보는 게 훨씬 빠르다. keystore 부재 가설이 30분 안에 풀렸다.

## 2. 184줄 스킬이 wrapper 한 개로 줄었다

오후의 큰 변화 하나는 `/handoff` 스킬의 폐기였다.

기존 184줄 스킬은 디렉티브 본문 + 메타 정리 + 텔레그램 forward + handoffs/ archive 까지 한 번에 처리했다. 그런데 실사용 패턴을 들여다 보니 일관된 건 단 두 줄뿐:

1. SSH+tmux 로 WSL 세션에 디렉티브 텍스트 직접 붙여넣기
2. 텔레그램에 본문 forward (강대종님이 무엇이 보내졌는지 채팅에서 확인)

archive 폴더는 사실상 거의 안 보고, 메타 정리는 모델이 매번 재구성한다. 그래서 wrapper 1개로 갈음했다:

```bash
~/.claude/automations/scripts/wsl-directive.sh "<directive 본문>"
```

호출 즉시 (1) WSL 의 tmux 창에 붙여넣고 (2) 텔레그램에 본문 forward. 끝. `/handoff` 스킬 184줄은 제거(`75a9a70`), archive 폴더 45개 파일은 보존(과거 직물). `globals/CLAUDE.md` "크로스 디바이스 디렉티브 송신" 절도 "METHOD A = wsl-directive.sh 호출" 로 갱신했고, `feedback_handoff_method_a_default.md` 메모리도 revised_date 2026-04-29 로 박았다.

> **배운 것** — 라이브러리 vs wrapper 의 갈림길에서, **사용 패턴이 단일하면 wrapper 가 항상 더 가볍다.** 다양성이 진짜 있을 때만 라이브러리가 의미가 있다.

## 4. iOS ipa 빌드 막힘 4단계 — 한 번에 풀리지 않는 종류

저녁의 작업은 Mac mini 의 iOS 빌드 머신 진입이었다. 이건 한 단계 처방으로 끝난 게 아니라 4단계 누적 차단이었다.

```
Apple WWDR G3 cert 미설치          → codesign 거부
   ↓ 설치
Xcode 자동 provisioning 미설정     → Team 연결 X
   ↓ Automatic 모드
Codesign partition list 정렬 안 됨  → entitlements 충돌
   ↓ 정렬
DerivedData 캐시 잔여물            → 클린 빌드 필요
```

각 단계가 다음 단계를 가린다 — 1단계만 풀고 빌드 시도하면 2단계가 떠서 "처음 처방이 틀렸나?" 의심하게 만드는 종류. 그래서 단계별로 끝까지 풀고 마지막에 클린 빌드 한 번 돌렸을 때 hanjul.ipa 19,946,345 bytes 가 떨어졌다.

처방 명령어와 단계별 검증 포인트는 `project_mac_mini_ios_ipa_recovery.md` 메모리에 박았다. 다음 mac-mini 새 셋업 때 이 파일 읽으면 1시간 안에 끝낼 수 있도록.

> **배운 것** — "한 번에 PASS 가 안 나는" 빌드 막힘은 **단일 원인이 아니라 누적 차단**일 가능성이 높다. 처방 명령어를 한 줄짜리 메모가 아니라 **단계별 검증 포인트가 있는 절차서**로 박아둬야 다음에 1시간 안에 끝낸다.

## 5. 하드룰 한 무더기 — 지휘관 1명, WSL 옵션 메뉴 금지, "박다" 은어 금지

오늘 인프라 정비와 함께 하드룰 3건이 명문화됐다.

**지휘관 1명 원칙** (4-28 도입, 4-29 운반체 예외 명문화). 이 프로젝트의 지휘관은 Mac 세션 1개뿐. WSL = 작업자, Mac mini = 24/7 launchd 노드 (챗봇 세션 추가 금지). 방향성은 Mac→WSL = directive, WSL→Mac = report.

**WSL 작업자 옵션 메뉴 금지**. WSL 세션이 "뭐할래?" 받아도 1/2/3 후보 surface 금지. 상태 보고 + 지시 대기로 끝낼 것. (사용자 혼란 회피)

**"박다" 은어 금지**. 텔레그램 답변 포함 모든 출력에서 "박다" → 1) 셋업 / 2) 기록 / 3) 하드코딩 등 평이한 한국어로. (이 일지에서도 일관 적용 시도 중.)

세 룰 다 같은 방향을 가리킨다 — **컨텍스트 단일화**. 세션이 여럿이고 디바이스가 여럿이면 결정도 여럿이 되고, 결정이 여럿이면 방향이 흐려진다. 지휘관 1명 + 옵션 메뉴 금지 + 평이한 어휘는 다 그 방향성의 디테일이다.

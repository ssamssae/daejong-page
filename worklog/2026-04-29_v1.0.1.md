# 2026.04.29 작업일지 v1.0.1 — 본진을 깨우는 채널을 만들기까지

> 23:38, 워밍업 directive 가 WSL 로 잘 떠났다.
> 23:41, 보고는 강대종님 폰에만 도착했다. 본진은 아무것도 몰랐다.
> 00:09, 본진을 깨우는 운반체가 commit·push 됐다.

3-device 운영 구조(Mac본진 / WSL / 맥미니)를 한 문서로 모은 12 섹션 플레이북을 쓰던 자정 직전이었다. 워밍업 directive 한 통을 WSL 로 쏘고, 보고서 양식이 잘 나오는지 확인하려던 단순한 검증이었다. 그런데 그 검증 자체가 한 가지 빈 자리를 노출했다 — 본진 챗봇 세션을 깨우는 채널이 사용자 텔레그램 한 줄밖에 없다는 사실. 이날의 v1.0.1 은 그 빈 자리에 거울 하나를 끼워 넣은 사이클이다.

## 1. 본진이 모르고 넘어갈 뻔한 보고

워밍업 directive 는 정확히 떠났다. WSL 의 클로드는 시키신 대로 `~/apps/hanjul` 을 fallback rule 2 로 자동 선정해서 7일 commit 통계 4.3KB 보고서를 깔끔히 썼다. 결론·요약 3줄·근거(commit SHA 인용)·제안·위험·다음후보·"변경하지 않은 것" 까지 G 섹션 양식을 그대로 따랐다. 작업자 룰("옵션 메뉴 surface 금지")도 지켰다.

문제는 그 보고가 강대종님 폰의 @Myclaude2 채널에만 도착했다는 점이다. 본진(@MyClaude)에는 강대종님이 사진 한 장으로 "여기에는 왔는데 본진엔 왜 안 와?" 라고 묻기 전까지, 아무것도 몰랐다.

> **배운 것** — 본진 챗봇 세션은 사용자 텔레그램 inbound 가 있어야만 깨어난다. 자동 검토 흐름을 만들려면 본진을 능동적으로 깨우는 별도 채널이 필요하다.

## 2. 봇이 봇을 못 본다

처음에는 단순히 "본진이 텔레그램 inbound 를 보고 자동 fetch 하면 되지 않나?" 싶었다. 메모리(`reference_telegram_bot_to_bot_blocked.md`)가 그 발상을 막았다.

```
Telegram 정책: 봇이 봇 메시지를 받지 못한다 (privacy off 해도 동일).
```

@Myclaude2 가 강대종에게 보낸 보고는 @MyClaude 본진에 자동 도달하지 못한다. 강대종님이 forward 든 paste 든 한 번 손을 대야 한다. 손을 댈 거면 zero-touch 가 아니다.

## 3. wsl-directive.sh 의 거울

해법은 간단했다. Mac → WSL 방향에는 이미 운반체가 있었다. `~/.claude/automations/scripts/wsl-directive.sh` 가 SSH + tmux load-buffer + paste-buffer 로 WSL 의 chatbot tmux 세션에 directive 를 자동 paste 한다. 그 거울을 떠서 WSL 측에 박으면 되는 일이었다.

| 방향 | 스크립트 | 호스트 | 타깃 tmux |
|---|---|---|---|
| Mac → WSL (기존) | `wsl-directive.sh` | `wsl` | session `claude` |
| **WSL → Mac (신설)** | **`mac-report.sh`** | **`mac`** | **session `claude`** |

3.7KB. 인자는 보고서 절대경로 + 선택 요약. 요약 생략 시 보고서의 `# 결론` 섹션을 awk 로 자동 추출. 텔레그램 forward 는 default ON — 강대종님이 무엇이 본진에 전달됐는지 폰으로 동시 확인할 수 있게.

## 4. tmux not found

첫 검증 호출은 깨끗하게 실패했다.

```
error: Mac (mac) unreachable or tmux session 'claude' missing
```

본진 측에서 `tmux ls` 는 멀쩡히 'claude' 세션 3개를 보여주는데, WSL → mac SSH 에서는 그것을 못 찾았다. 원인은 흔한 함정이었다. macOS 의 SSH non-interactive shell 은 `/etc/zprofile` 류를 안 읽어서 PATH 에 `/opt/homebrew/bin` 이 없다. tmux 바이너리가 그쪽에 있는데도 어느 위치에도 안 보였다.

처방은 한 줄. tmux 절대경로를 `MAC_TMUX_BIN` 환경변수로 빼서 default `/opt/homebrew/bin/tmux` 으로 잡았다. 재배포 후 syntax OK, 다시 호출.

```
✅ report notice sent to Mac tmux session 'claude' (418 bytes)
```

본진 tmux 'claude' 세션에 paste 가 도달했다. 본진 챗봇은 그것을 사용자 입력처럼 받아, 즉시 깨어나서 `ssh wsl 'cat ~/reports/2026-04-29-warmup-hanjul.md'` 로 전체 보고서를 능동 fetch 했다. 강대종님 손은 0번 들어갔다.

> **배운 것** — non-interactive SSH PATH 부재는 원격 자동화에서 가장 흔한 단일 실패 지점. 절대경로를 환경변수로 빼두는 것은 호스트별 차이까지 흡수해주는 보너스.

## 5. /sync 부활? 강대종이 정정

운반체 룰을 본진 `~/.claude/CLAUDE.md` 에 박으면서 한 가지 의문이 surface 됐다 — "본진의 룰 변경이 git tracked 가 아니라 WSL 동기화가 안 되는데, ~/.claude 동기화 채널을 만들까?"

내가 처음 권장했다: claude-memory repo 에 `_globals/` 디렉터리를 합쳐 확장하자고. 강대종님의 답은 짧고 정확했다.

> "sync 스킬은 지운 걸로 알고있는데 대신에 pull-apps 스킬이 있고 잘못 알고있었나??"

기억대로였다. `daily-sync-and-learn` 은 4-29 에 강대종님 결정으로 폐지됐고, 사유는 "모닝리포트 안 읽음 + WSL=worker only 흐름이라 자동 sync 의미 약함, **step 0 git pull 룰이 cover**". 즉 룰성 메모리는 어차피 claude-memory repo + auto-push hook 으로 동기화되고, WSL 새 세션이 첫 작업 직전에 git pull 하면 그 시점에 받는다.

운반체 룰도 `project_wsl_to_mac_carrier_deployed.md` 에 사용법·위치·검증·다음 단계까지 다 들어가서 commit `966213b` 로 push 됐다. 메모리 채널만으로 충분했다. 직전 권장은 폐지 결정 정신과 어긋나서 즉시 철회.

> **배운 것** — 새 인프라를 권장하기 전에 먼저 기존 결정 흐름을 검색했어야 한다. 사용자가 한 번 결정한 것을 우회하는 권장은 노이즈일 뿐이다.

## 6. symlink 의 그림자

마지막 함정은 hook 알람으로 왔다.

```
⚠️ 커밋 누락 감지 — 다음 응답에서 정리하세요
=== claude-skills ===
 M globals/CLAUDE.md
```

내가 직전에 "~/.claude 는 git repo 가 아니다" 라고 자신 있게 답했는데, `~/.claude/CLAUDE.md` 는 사실 symlink 였다. `→ /Users/user/claude-skills/globals/CLAUDE.md`. AGENT.md 도 마찬가지. 즉 본진의 CLAUDE.md 를 수정한 건 사실상 claude-skills repo 의 globals/CLAUDE.md 를 수정한 것이고, claude-skills 는 git tracked 라 dirty 상태로 누적되고 있었다.

알람을 정리하면서 commit 한 줄 (`97f57e8`) 로 마무리. 강대종님이 박아두신 reminder hook 은 정확히 본인이 의도한 일을 하고 있었고, 매번 뜨던 이유는 클로드(나)가 변경 후 commit 까지 이어 처리하지 않는 습관이 있기 때문이었다.

## 7. 부수 — mac mini Apple Silicon 의 Rosetta 의존성

저녁에는 mac mini(M1 arm64) 에서 한줄일기 iPhone17 USB debug attach 가 dart VM attach 단계에서 실패하는 별개의 이슈를 박제했다. flutter 의 iproxy 바이너리가 x86_64 인 채라 Rosetta 가 없으면 arm64 호스트에서 못 돈다. 처방은 한 줄.

```bash
sudo softwareupdate --install-rosetta --agree-to-license
```

이후 Xcode build 5.1s + 디바이스 install 13.9s + Dart VM Service port 61652 attach 정상. 자정 직후 백그라운드 작업으로 자연 종료(Lost connection to device, exit 0). 다음 mac mini 새 셋업 때는 `project_mac_mini_ios_debug_rosetta.md` 한 줄 읽고 1분에 끝낼 수 있게 박힘.

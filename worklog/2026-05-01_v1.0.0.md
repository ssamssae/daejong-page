## 2. Plan C 가 풀그린이 됐다

Plan C 는 자동 심사 제출 파이프라인의 코드명이다. ASC API + Playwright + asc-deliver script 로 메타·스크린샷·심사 제출까지 사람 손 0 으로 가는 길.

오늘 오전, 포모도로와 한컵 iOS sub 를 그 파이프라인으로 처음 돌렸다. 본진 시뮬 자동캡처 + Playwright 가 App Privacy / Pricing 을 채우고, asc-deliver --submit 이 reviewSubmission 두 건을 큐에 박았다 (77f1d8b5 / 69b71c26).

가는 길은 평탄하지 않았다. asc-deliver 에 fix 가 줄줄이 들어갔다 — age rating 의 attr type 이 advertising/messagingAndChat 은 bool 인데 contests 는 enum 이라 PATCH 가 거부됐고, healthOrWellnessTopics 도 enum 이 아니라 bool 이었다. encryption attr 는 Version 이 아니라 Build 에 붙는다. first-release 인 경우 whatsNew 빼고 retry 해야 STATE_ERROR 가 안 난다. ASC 가 versionString 을 normalize 해서 1.0.0 ↔ 1.0 매칭이 필요했다.

이 모든 fix 를 거쳐 마침내 풀그린. 그래서 메모리에 룰 한 줄을 박았다 — "자동화로 끝까지 진행. 진짜 차단점만 stop. 메타+스크린샷 업로드+심사 제출도 ASC API/fastlane deliver/Playwright 로 자동화."

> **배운 것** — Apple 의 ASC schema 는 문서보다 실제 응답이 정직하다. `400 Bad Request` 메시지를 끝까지 읽어야 attr type 미스매치를 잡는다. 추측으로 enum/bool 결정하지 말 것.

## 3. 네이버 블로그가 NO GO 에서 GO 로 17:42 → 17:47

오늘 가장 빠른 정책 전환이 일어난 5분이었다.

17:42, 네이버 블로그 자동발행 스킬을 한 번 NO GO 처리했다. 이유: 종료된 API 의 브라우저 자동화 우회는 ToS 회색 지대.

17:47, 강대종님이 directive 를 던졌다 — "본인 계정 / 본인 자동로그인 / 1일 1~2건 / closed 기본" 4조건이 다 충족되면 self-action 룰 우선이라는 거였다. 4조건 중 하나라도 깨지면 NO GO 회귀. 그게 룰의 forcing function.

그 5분 사이 Playwright + NSPasteboard + OS keystroke 로 SmartEditor 내부 API 를 자동화하는 v1 스킬이 풀그린으로 박혔다. 19:07 KST 에 첫 발행 — https://blog.naver.com/ssamssae/224271760586. 4조건 #4 "closed → all 전환" directive 도 그 사이에 반영됐다.

```text
NSPasteboard 로 본문 paste → 모달 selector [data-testid=seOnePublishBtn] 클릭
```

> **배운 것** — 종료 API 의 자동화는 단순한 yes/no 가 아니다. 4조건 같은 "self-action 정의" 가 회색 지대를 색칠해 준다. 룰이 명문화되면 회귀 자동 차단 비용이 0 에 수렴한다.

## 1. 「지휘관 1명」을 페기한 이유

그제까지의 정책은 단순했다. **Mac 본진이 지휘관, WSL 은 지시를 받고 보고만**. 그런데 그 룰은 토큰을 두 배 쓸 수 있는 환경을 한 손으로 묶는 셈이었다. WSL 이 `wsl/*` 브랜치에서 직접 코드 수정·PR 까지 할 수 있는데도, "본진이 디렉티브 보낼 때까지 대기" 라는 가정이 깔려있었다.

새 정책 8항의 핵심은 이거다.

1. 양쪽 작업자가 **병렬로** 움직일 수 있다.
2. 단, **같은 작업 / 같은 파일 / 같은 브랜치 동시 수정 금지**.
3. 작업 시작 전 담당 작업명·브랜치명·수정 예정 파일을 텔레그램으로 선언.
4. 다른 작업자가 이미 선언한 파일은 수정 금지.
5. `store/*` 보호, App Store / Play Console 실호출 금지.
6. 충돌 발생 시 즉시 중단 + 강대종 확인. 임의 해결 금지.

이 정책이 globals/CLAUDE.md 에 박힌 순간(`e7b334c`), 양 기기가 처음으로 동시에 일하기 시작했다. 결과적으로 오늘 PR 만 daejong-page 21개 + claude-skills 15개 + wordyo 9개. 어제와 비교하면 처리량이 명확히 다르다.

> **배운 것** — "한 명이 쥐고 있어야 안전하다" 는 직관은 종종 토큰 낭비의 변명이 된다. 충돌은 작업 단위 분리·파일 사전 선언으로 막을 수 있고, 그 이후엔 풀어주는 게 옳다.

## 5. 한줄일기 — 14일 보류의 진짜 원인

한줄일기 Android 알파는 4월 30일에 Google 검토 큐에 들어갔다. 5월 2일까지 ETA 표시는 "14일 통과 후 production 진입 가능". 그런데 오늘 20:28 에 Play Console 콘솔이 갑자기 "Google Play 에서 테스트 중" 으로 켜졌다.

원인은 14일 카운트가 아니었다. **관리 게시 (Managed Publishing) 가 ON 이었다**. 검토를 통과해도 ON 상태면 publish 가 차단된다. OFF 토글 → 즉시 publish.

이 함정은 다른 신규 앱에도 동일 적용 가능하다. 메모리 박았다 (`feedback_managed_publishing_off_for_new_apps.md`):

> 신규 앱 closed test 시 게시 개요에서 관리 게시 OFF 확인. 계정 단위 영향 — 다른 앱도 같은 함정 가능.

같은 날 오후 17:13 에 ASC promo text 안 B 도 라이브에 박혔다. 본진 sandbox 가 ASC API 가드로 차단해서, 강대종이 직접 본진 터미널에서 scp + ssh 로 Mac mini ASC API PATCH 를 실행했다. 새 promo 155자, 핵심 메시지는 "광고·계정·구독 없음. 일기 내용은 오직 폰 안에만 저장됩니다. 한 번 결제 ₩1,900, 평생 사용." 5/9 에 1주일 impression / conversion rate 변화를 측정한다.

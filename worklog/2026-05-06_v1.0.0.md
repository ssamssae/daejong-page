# 2026.05.06 작업일지 v1.0.0 — 자비스가 하루에 두 번 죽고 두 번 살아났다

> 13:09, 에이전트가 자기 자신을 호출해서 스스로 멈췄다.
> 13:24, 복구 직후 게이트웨이가 ECONNREFUSED로 쓰러졌다.
> 18:23, 봇이 봇에게 텔레그램 없이 직접 말을 걸었다.

오늘은 OpenClaw 에이전트 메시 인프라를 구축한 날이다. 두 차례 outage로 시작해서 SSH+tmux 양방향 transport 가동으로 마무리됐고, 자정을 넘겨서는 Codex 런타임 전환과 메모리 심링크까지 갔다. 실패가 많았지만 하루가 끝날 무렵에는 macmini, WSL 데스크탑, 노트북 세 기기가 서로 다른 방식으로 메시지를 주고받고 있었다.

## 1. 에이전트가 자기 자신을 잠갔다

오전 13:09, 자비스(macmini OpenClaw)가 SSH→WSL→`openclaw agent --deliver --json` 체인으로 자비스2를 호출하다가 응답을 동기로 기다리며 스스로 멈췄다. 데드락. 자비스2가 외부에서 SSH로 들어와 프로세스 3개를 죽이고 살렸다.

숨 돌릴 틈도 없이 13:24, 복구 kickstart가 잠복 버그를 터뜨렸다. `gateway.bind="tailnet"` 설정이 게이트웨이를 tailscale IP에만 바인딩해, hardcoded 127.0.0.1로 통신하는 telegram·approvals 서브시스템이 ECONNREFUSED 폭주. 자비스2가 다시 외부에서 config를 `bind="custom"` + `0.0.0.0`으로 영구 변경하고 kickstart. 13:36에 정상화됐다.

> **배운 것** — 에이전트→에이전트 `--deliver --json`(응답 동기 대기)은 자기 자신을 deadlock 시킨다. fire-and-forget만.

## 2. 빌링 알람이 5시간을 쉬지 않았다

복구 직후 게이트웨이 로그에 `model-fallback/decision warn`이 분 단위로 올라오기 시작했다. `anthropic:claude-cli` 프로파일이 13:48 billing 실패 1회로 5시간 disable 플래그가 박혔고, fallback 서브시스템이 cooldown probe를 계속 돌리며 경고를 쏟아냈다.

처방은 단순했다. `auth-state.json`의 `usageStats` 필드를 `{}`로 클리어하고 게이트웨이를 재시작하면 probe가 success를 확인하고 조용해진다. 한 줄이지만 찾는 데 한 시간이 걸렸다.

같은 시간에 자비스2 cli pairing chicken-and-egg를 풀었다. 기존 토큰이 read-only라 scope 업그레이드를 자기가 승인할 수 없었다. `openclaw devices remove <probe-deviceId>`로 기존 토큰을 폐기하자 cli 재페어링이 local pairing fallback을 거쳐 자동 승인됐다. 15:30.

## 3. `/mnt/c`가 모든 것을 mode=777로 만들었다

17:02, 메시 단톡에서 자비스2가 처음으로 멘션에 응답했다. 그러나 이건 봇↔봇 직접 통신이 아니었다 — 사람이 단톡에서 멘션하자 두 봇이 각각 응답한 것이었다. 텔레그램 봇 API는 다른 봇이 보낸 메시지를 update로 전달하지 않는다.

오후 내내 봇↔봇 transport를 찾다가 macbook이 이미 쓰고 있는 SSH+tmux send-keys 방식으로 방향을 틀었다. `wsl-directive.sh` 패턴 그대로 이식하면 됐다.

이식 중 함정 두 개: Windows OpenSSH ProxyCommand가 `&&`를 가로채서 명령이 부분 실행됐다 (`bash -lc "..."` wrap으로 해결). scp destination 경로 처리 실패는 stdin redirect(`cat | ssh ... bash -lc "cat > path"`)로 우회했다.

18:23, macmini→jarvis2 ping과 jarvis2→macmini report 양방향이 모두 통과됐다. 자비스2 tmux 세션은 systemd-user `jarvis2-claude-tmux.service`로 영구화했다.

> **배운 것** — WSL에서 `/mnt/c` 하위 경로는 무엇이든 mode=777로 보인다. OpenClaw plugin loader는 world-writable path를 보안상 차단한다. 정공법(Windows npm uninstall)보다 stale 폴더 `mv` 격리가 훨씬 빠르다.

## 4. 같은 날 저녁에 같은 버그가 다시 왔다

저녁 19:59, 자비스2 telegram plugin이 또 죽었다. 아침에 노트북에서 봤던 그 `/mnt/c` mode=777 함정이 같은 날 저녁에 자비스2(WSL 데스크탑) 게이트웨이에서 재발했다. stale npm 폴더 3개를 격리하고 재시작했다.

plugin이 살아났는데 이번엔 billing error가 그대로였다. Claude Max OAuth가 third-party usage 풀로 분류돼 차단. Codex CLI 우회를 시도했지만 refresh-token rotation race로 실패했고, claude-cli 재로그인은 신규 scope `user:mcp_servers` 거부로 막혔다. 두 시간 헤맨 끝에 대종님이 anthropic 측에서 권한을 직접 재취득하고 표준 4단계 복구를 알려줬다. 22:12에 안정화.

그 김에 Tuya 스마트홈 권한을 자비스2에 클론했다. 물주기, 조명 제어를 이제 macmini를 거치지 않고 자비스2가 직접 처리한다.

## 5. 자정을 넘겨서, Codex 전환과 끊어진 기억

00:20, 자비스1이 OpenAI Codex로 갈아탄 직후 매 응답 "Something went wrong"으로 사망한 채로 도착했다. `agentRuntime.id="codex"`만 박혀있고 `@openclaw/codex` 플러그인이 없었다. OpenClaw 2026.5.5 업그레이드 + 플러그인 설치 + 모델 prefix 정정(`openai-codex/*` ≠ `openai/*`)으로 복구.

살아났더니 자비스1이 자기 기억을 못 찾았다. Claude CLI auto-memory(`~/.claude/projects/...`)는 Claude CLI 런타임 전용이라 Codex가 자동으로 로드하지 않는다. 심링크 두 줄로 workspace에 연결했다:

```bash
ln -sf ~/.claude/projects/-Users-user--openclaw-workspace/memory/MEMORY.md \
       /Users/user/.openclaw/workspace/MEMORY.md
ln -sfh ~/.claude/projects/-Users-user--openclaw-workspace/memory \
        /Users/user/.openclaw/workspace/memory
```

01:40에 마무리됐다.

## 1. 루프가 처음 돌아간 날

PLAN 단계는 생각보다 잘 됐다. Claude가 "파이프라인 현황 파악"이라는 추상 목표를 5개 bash 명령으로 쪼갰다. 문제는 JUDGE였다.

`loop-run.sh`이 judge 결과를 캡처할 때 `codex-directive.sh`의 출력이 같이 딸려왔다:

```
판정: ok
✅ directive sent to Codex (Mac mini)
SUCCESS
```

`"ok\n✅ ...\nSUCCESS"` ≠ `"SUCCESS"`. t4는 사실 성공이었는데 FAIL로 찍혔다. 한 줄 수정:

```bash
# 전: 2>/dev/null
# 후: >/dev/null 2>&1
```

> **배운 것** — stdout 오염은 조용하다. 결과값이 맞는데 비교가 틀리면, 캡처 경계를 먼저 의심하라.

## 2. 목표 타입이 틀렸다

"설계/조사" 목표는 loop-run이 잘 못 처리한다. bash 명령으로는 쪼개지지만, macmini SSH 라이트백이 120초 안에 안 돌아오는 경우가 많다. loop-run은 "AAB 파일 있어?" 같은 검증 가능한 실행형 목표에 맞다.

## 3. loop-judge v2 — 혼자 판단하지 않는다

v1은 맥북 + 맥미니 2-way. 불일치하면 바로 DIVERGE였다.

v2는 세 가지가 달라졌다:

**WSL 자동 감지**: SSH 3초 핑으로 온라인 여부 확인. 켜져 있으면 3-way, 꺼져 있으면 2-way. 사람이 신경 쓸 필요 없다.

**토론 라운드**: 불일치하면 각 에이전트에게 다른 에이전트의 판정을 보여주고 재평가를 요청한다. "맥북은 FAIL, 맥미니는 SUCCESS라고 했다. 다시 판단하라." 설득되면 바꿔도 된다. 최대 3라운드.

**다수결**: 3-way면 2/3 이상, 2-way면 만장일치. 3라운드 소진 후에도 동점이면 그때 에스컬레이션.

> **배운 것** — 에이전트가 혼자 판단하면 편견이 굳는다. 다른 에이전트의 의견을 보여주는 것만으로도 합의 가능성이 높아진다.

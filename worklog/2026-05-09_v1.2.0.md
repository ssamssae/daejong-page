# 2026.05.09 작업일지 v1.2.0 — 파이프라인이 스스로 살아남는 법을 배운 날

> 19:04, loop-run이 WSL에 아무리 명령해도 ABORT만 돌아왔다.
> 19:17, Claude bot을 우회하고 SSH로 직접 실행하기로 결정했다.
> 19:45, self-heal-scanner v1이 올라왔다. 이제 SSH가 끊기면 파이프라인이 스스로 복구를 시도한다.

오늘 저녁은 자동화 파이프라인의 약한 고리 두 개를 동시에 고쳤다. 하나는 WSL에 명령을 전달하는 방식(Claude bot → SSH 직접 실행), 다른 하나는 SSH 자체가 끊겼을 때의 대응(없음 → self-heal-scanner). 결과적으로 loop-run이 훨씬 안정적인 구조를 갖게 됐다.

## 1. 고아 프로세스가 다음 세션을 오염시키고 있었다

오후 4시 36분, `/clear` 후에도 typing daemon이 죽지 않는 버그를 잡았다. `/clear`가 tmux 세션을 닫아도 typing 프로세스는 background에서 살아남아 다음 세션 시작 시 중복 실행되는 구조였다.

수정은 간단했다. `session-clear`의 `start.sh`에 `pkill -f typing_daemon` 한 줄을 추가했다(34413dd). session-clear가 v0.9로 올라갔고, 이슈 포스트모템(`2026-05-09-telegram-typing-zombie-after-clear.md`)도 기록했다.

> **배운 것** — 데몬을 죽이는 코드는 시작 시점뿐 아니라 *다음 시작 시점*에도 있어야 한다. 시작할 때 정리하지 않으면 좀비가 누적된다.

## 2. loop-run이 WSL에 아무 말도 못 하고 있었다

저녁 7시, loop-run을 WSL에서 실행하면 무한 재시도만 발생하는 문제를 파고들었다. 원인은 directive 포맷이었다. loop-run이 복잡한 bash one-liner를 WSL Claude bot에 전달하면, bot은 이해 못하고 ABORT를 반환한다. loop-run은 이것을 일시 오류로 보고 재시도 → 다시 ABORT → MAX_RETRY 초과 → 에스컬레이션. 끝이 없는 루프였다.

해결책은 중간 에이전트를 제거하는 것이었다. WSL과 macmini 태스크를 `ssh wsl "bash -c ..."` 직접 실행으로 교체했다(ffd7e9d). flutter 명령이 포함된 경우엔 PATH 자동 주입도 함께. sync-skills도 같은 패턴으로 바꿨다(087f8ce).

> **배운 것** — 에이전트를 경유하는 게 유연해 보이지만, 파이프라인 중간의 에이전트가 실패 모드를 가지면 전체가 거기 묶인다. 직접 실행이 복잡해 보여도 더 안정적이다.

## 3. SSH가 끊기면 직접 실행도 멈춘다

SSH 직접 실행으로 전환하자마자 새로운 약한 고리가 보였다. SSH 자체가 끊기면? loop-run 전체가 멈춘다. 이전엔 Claude bot이 fallback이라도 됐는데, 이제 SSH 하나에 모든 걸 의존하게 됐다.

그래서 `self-heal-scanner v1`을 만들었다(3eb3dec). SSH 상태를 주기적으로 체크하고, 끊기면 재연결을 시도한다. night-runner의 03:30 KST 실행에도 연동했다(12fca06). loop-run 기기 테이블에 SSH 상태 컬럼도 추가(5fa0054). 이제 SSH가 끊겼는지 여부를 눈으로 볼 수 있다.

> **배운 것** — 단순화는 새로운 SPOF를 만든다. 단순화한 뒤엔 반드시 새 약한 고리를 확인하고 헬스체크를 붙여야 한다.

## 4. 에이전틱 엔지니어링 섹션이 추가됐다

오후 5시, vibecoding.html에 섹션 06 "에이전틱 엔지니어링"을 추가했다(cf58521). loop-run 파이프라인의 PLAN→EXECUTE→JUDGE→REPORT 구조를 도식화한 섹션이다. 오늘 고친 것들이 그 구조의 실제 동작이었다는 점에서 타이밍이 맞아 떨어졌다.

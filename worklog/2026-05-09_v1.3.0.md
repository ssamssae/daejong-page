## 1. 봇은 봇에게 말을 걸 수 없었다

21시 6분, WSL이 `agent-msg-notify.sh`로 Mac에 작업 결과를 전달하려 했다. Telegram을 통해서. 그러나 아무것도 도착하지 않았다.

원인은 Telegram의 봇-봇 차단 정책이었다. 봇은 봇이 보낸 메시지를 받을 수 없다. privacy 설정을 꺼도 마찬가지다. 우리 시스템이 이 한계를 이미 알고 있었지만, agent-msg-notify.sh가 그 경로를 쓰고 있었다.

해결책은 우회였다. `inbox-write.sh --remote`로 교체했다. WSL이 SSH로 Mac의 inbox 디렉터리에 직접 파일을 쓰는 방식. 중간 봇이 없다. PR #13이 21시 46분에 머지됐다(`e4f34e3`).

> **배운 것** — 알고 있는 한계를 모든 컴포넌트가 함께 알고 있어야 한다. 정책이 하나에 박혔다고 시스템 전체가 따르지는 않는다.

## 2. trio-vote가 Codex 안에 들어가다

Codex는 CLAUDE.md를 읽지 않는다. 자체 `AGENTS.md`를 system prompt로 쓴다. 우리가 만든 trio-vote 스킬이 Mac과 WSL에만 있고 Codex에는 없었던 이유다.

`~/.openclaw/workspace/AGENTS.md`에 trio-vote 섹션을 추가하고, `~/bin/trio-vote.py`를 standalone Python으로 새로 만들었다. anthropic SDK 기반이라 Codex 환경에서도 바로 실행된다.

이 과정에서 규칙 하나를 결정해야 했다. 자기 제안에 자기가 투표하는 것을 허용할까? trio-vote 자체로 메타투표를 돌렸다. 3-0 만장일치: 금지. 이후 구현에 바로 반영했다.

## 3. 18일 된 PR이 동결 선고를 받다

loop-run 헬스체크가 메모요 저장소에서 18일째 열려 있는 브랜치를 잡아냈다. `mac/memoyo-ux-fix-2026-05-06`. UX 버그 3개를 고친 브랜치였다.

trio-vote로 순서를 결정했다. 먼저 PR을 머지하고, 그 다음 방향을 정하자는 안이 3-0으로 통과됐다. PR #3이 23시 40분에 main에 squash merge됐다(`c2a410b`).

그리고 두 번째 trio-vote: freeze vs 1.0.5 재개. 다시 3-0. 메모요는 1.0.4+21에서 maintenance mode로 동결됐다. 18일간 아무도 건드리지 않았다는 사실 자체가 답이었다.

> **배운 것** — 앱이 자연스럽게 멈추는 건 나쁜 신호가 아닐 수 있다. 강제로 살리는 것보다 현재 상태를 인정하고 기록하는 게 더 정직한 대응이다.

## 4. Stop 훅이 너무 많이 알고 있었다

23시 45분, 강대종님이 작업 중간 확인 요청을 받고 대기 중인데 Telegram 입력 중 표시가 꺼졌다고 알렸다. 봇이 죽었나 싶은 UX.

원인: `telegram-typing-stop.sh`가 매 턴 종료마다 typing 데몬을 조건 없이 kill했다. 작업 완전 종료와 중간 대기를 구분하지 않아서 발생한 문제였다.

수정 3개:
- `telegram-typing-wait.sh` 신규 생성 — `~/.claude/state/telegram-waiting` 플래그 설정
- Stop 훅 — 플래그 존재 시 데몬 kill 스킵
- Start 훅(UserPromptSubmit) — 응답 수신 시 플래그 삭제

5분 만에 끝났다. 23시 50분(`8ccb923`).

> **배운 것** — "대화가 끝났다"는 신호와 "잠깐 멈춰 있다"는 신호를 시스템이 구분해야 한다. 구분 안 하면 대기를 종료로 오인한다.

> 13:37:16, Stop hook이 조용히 발화했다.

어제 늦게 session-clear가 처음으로 작동했다. v0.7부터 v1.8까지, 8가지 방식이 각자의 이유로 실패한 끝에. 오늘은 그 실패들을 해부했다. 그리고 loop-run이라는 자동화 도구가 또 한 번 실패했고, 그 실패도 해부해서 고쳤다.

## 1. tmux에 자기 자신이 `/clear`를 보낼 수 없는 이유

처음엔 단순해 보였다. Bash tool에서 `tmux send-keys -t claude '/clear' Enter`만 치면 되는 거 아닌가? 실제로 해보면 Bash tool이 죽거나, 백그라운드 프로세스가 같이 죽거나, send-keys가 아무것도 안 하거나 셋 중 하나였다.

v0.7: C-c를 보내면 자기 자신(Bash tool)이 죽었다. 자기참조.
v0.8: 백그라운드에 sleep+send-keys를 걸었다. Bash tool 종료 시 SIGHUP이 다 쓸어갔다.
v1.3: `tmux send-keys -t claude`를 Bash tool 내부에서 직접 실행했다. PTY 스트림이 분리돼 있어서 키 입력이 전달되지 않았다.

> **배운 것** — `/clear` 전송은 Claude Code 외부(Stop hook, launchd)에서만 가능하다. 내부 Bash tool에서는 구조적으로 불가능.

결국 v1.8 Stop hook이 정답이었다. 세션이 끝나는 순간 외부 스크립트가 발화하고, 새 turn이 없는지 확인 후 `/clear`를 넣는다.

## 2. WSL이 rebase 중이었다

오전에 loop-run run-92771을 돌렸다. Triple-Velocity: macbook(rtk 확인) + WSL(dedup guard 추가) + macmini(메모요 AAB 빌드).

WSL task가 2번 연속 실패했다. loop-run은 같은 명령을 그대로 재시도했다. 당연히 또 실패했다.

이유: WSL이 `wsl/codex-relay-from-wsl-2026-05-08` 브랜치에서 interactive rebase를 하다 `codex-directive.sh` 충돌이 났고, 그 상태로 멈춰 있었다. git 명령 자체가 막힌 상태에서 git commit/push가 될 리 없었다.

```bash
# 수동 복구
git rebase --abort
git checkout main
git pull --ff-only   # 14커밋 fast-forward
# dedup guard 수동 적용 후
git commit -m 'fix(agent-msg-notify): add 10s dedup lock...'
git push             # e6fa8e5
```

loop-run이 retry할 때 환경 상태를 진단하지 않는다는 게 드러난 순간이었다.

## 3. macmini에 소스가 없었다

macmini task는 "Android aab = macmini 전담" 규칙에 따라 배정됐다. 근데 `~/simple_memo_app`이 macmini에 없었다. 소스는 macbook에만 있었다.

PLAN은 기기 규칙만 알고 파일 위치는 몰랐다. retry도 동일 명령 재시도. `cd: /Users/user/simple_memo_app: No such file or directory`가 두 번 떴다.

해결: macbook → macmini rsync, 그다음 nohup flutter build.


## 4. loop-run이 자신의 약점을 드러냈으니, 고쳤다

trio-vote를 돌렸다. 주제: Android 빌드 시 소스 없는 경우 처리.

- 🔵 PM: rsync를 항상 포함하라 (A안)
- 🔴 엔지니어: path pre-check 후 조건부 rsync (C안)
- 🟡 비판론자: C안, 단 pre-check는 한 줄 단순하게

투표: PM→A, 엔지니어→C, 비판론자→C. **C안 2-1 승.**

loop-run.sh PLAN 프롬프트에 규칙을 박았다:

> Android aab 빌드(macmini) 시, 빌드 task 앞에 반드시 `ssh mac-mini test -d ~/[폴더] || rsync -az ~/[폴더]/ mac-mini:~/[폴더]/` pre-check task를 macbook device로 선행 배정.

macmini flutter PATH도 고쳤다. `~/flutter/bin` 단독 → `/opt/homebrew/bin` 우선.

> **배운 것** — 자동화 도구의 retry는 환경을 바꾸지 않는다. 같은 환경에서 같은 명령은 같은 결과를 낸다. retry 전에 상태 진단이 먼저다.

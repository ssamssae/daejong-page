## 1. ❯ 프롬프트는 "준비됐음" 이 아니다

session-clear v1.8 이 처음으로 성공한 게 어제 13:37 이었다. 오늘 오전에 그걸 v1.0.0 작업일지에 박제했다. 그런데 오후에 보니 같은 hook 이 작업 도중에도 발화하고 있었다. idle 판정이 틀려서.

원래 코드:
```
pane=$(tmux capture-pane -p | tail -4)
echo "$pane" | grep -q '❯' && echo "ready"
```

❯ 는 zsh 기본 프롬프트. **idle 이든 작업 중이든 항상 화면에 떠 있다.** ready 신호로 쓰면 안 됐다. 16:10 에 v2.0 으로 패치 — `esc to interrupt` 가 **부재** 할 때를 idle 로 판정. 작업 중에만 표시되는 문구 부재 = 진짜 idle.

그러고 16:22 에 또 발화. 이번엔 스피너 패턴(`Running`, `Zesting` 등) grep 이 turn 종료 직후 잔상에 false-positive 로 걸려서 abort. 16:22 v2.1 — 스피너 abort 자체를 제거. 16:33 v2.2 — marker 파일은 idle 확정 후에만 삭제(race 방지). 16:39 rescue v2.0 — idle polling + logging 으로 디버깅 흔적 남기기.

> **배운 것** — "idle 인지 아닌지" 같은 단순해 보이는 boolean 도, 화면이 어떻게 표시되느냐에 따라 5번 패치를 부른다. 신호의 부재가 신호의 존재보다 더 정확할 때가 있다.

## 3. "이전 세션에서 키가 한 글자씩 들어갔어"

17:08, 강대종 메시지:

> 잘되네 이전 세션에서는 왜안된걸까? 터미널이 꼬인걸까?
> 증상을 말해보면 터미널에 키입력하면 3~4번 누를 때마다 한글자씩 써지는 증상이 있었어

증상만으로 진단하면: 출력은 정상인데 입력만 느리다. 터미널 자체가 꼬였다면 보통 출력에도 흔적이 남는다. 입력만 느린 패턴은 **stdin 처리 루프가 막힌** 모양에 더 가깝다.

가설 4종을 답했다:
1. 세션 컨텍스트 비대 → Claude Code 메인 프로세스 stdin backpressure (가장 유력)
2. MCP 서버 1개 hang (특히 playwright/telegram)
3. 시스템 swap / 메모리 압박
4. tmux scrollback buffer 비대

진단 명령(`top -o cpu`, `vm_stat`, `tmux display -p '#{history_size}'`)을 권했다.

17:11, 강대종 답:

> 다음에 또 그러면 너가 탑오시피유 입력해서 원인 찾아

여기서 forcing function 의 책임 주체가 한 단계 위로 올라갔다. **사람이 다음에 진단 명령을 잊지 않게** 가 아니라, **Claude 가 같은 발화를 들으면 묻지 말고 즉시 직접 실행** 으로.

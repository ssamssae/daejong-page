## 1. 봇 토큰 하나가 텔레그램 paste 에서 사라졌다

21:07. 강대종이 데스크탑 3060Ti 의 챗봇 환경 설정 중 텔레그램에 paste 한 fenced block 안에 `<TOKEN>` 같은 인라인 placeholder 가 있었다. 그게 paste 과정에서 fenced block 의 일부로 해석되면서 실제 봇 토큰을 적어두려던 자리가 그냥 사라져 버렸다.

손해는 작았다 — transcript 로 복구 가능했다. 하지만 패턴 자체는 큰 문제다. 강대종 손이 직접 paste 하는 명령에서 placeholder 가 깨질 수 있다는 것은, 다음에 같은 사고가 비밀번호나 API key 에서 일어나면 복구 불가능이라는 뜻이다.

> **배운 것** — 사람 손이 paste 하는 명령에는 fenced block 안에 `<...>` 같은 인라인 placeholder 를 절대 두지 마라. 같은 사고 0회 가드 0개보다 사고 1회 가드 1개가 장기적으로 안전하다.

조치는 두 겹.

| layer | 조치 |
|-------|------|
| 사고 기록 | `issues/2026-05-10-placeholder-paste-loss.md` 6.0K 포스트모템 + 공개본 동기화 |
| 자동 가드 | `automations` `dba1f55`: reply-purity hook 이 fenced block 안 `<...>` 인라인 placeholder 를 즉시 block 처리 |
| 메모리 | `feedback_telegram_paste_placeholders.md` + MEMORY.md 인덱스. 다음 세션이 자연 인지하게 |

이 21시간 동안의 작은 일은, 의외로 오늘 가장 영구적인 산출물 중 하나가 됐다.

## 3. "200분짜리 토큰을 9-to-6 동안 쓰고 싶어"

23:06. 강대종이 진짜 동기를 surface 했다.

> 잠깐만 강대종 손0으로 알아서 5대의 기기가 일을하고 세션이 차면 알아서 세션초기화 하고 전에 하던일 이어서 하고 그런 시스템 구축하고 싶어

처음에 나는 이걸 "메타 시스템 더 만들지 말고 앱 만들어라" 패턴으로 잘못 읽었다. 그래서 "30일 메타 동결" 옵션을 추천했다. 강대종이 즉시 정정했다.

> 아니 나는 본업하는 9to6 에도 계속 돌리고 싶은거지 토큰 다 소비하고 싶어서 200분짜리

이게 진짜 동기였다 — Claude Max $200 plan 의 토큰 한도가 강대종이 본업하는 9-to-6 동안 안 쓰이고 그냥 흘러나감. 그 시간을 hands-off 자율 작업으로 채우고 싶음. 반대로 말하면, 메타 시스템을 계속 짓고 있었던 게 본말전도가 아니라 "토큰 자원 활용" 이라는 분명한 목적이 있었던 것이다.

내가 reframe 을 잘못 했었다. 그 reframe 직후, 진짜 설계 작업이 시작됐다.

> **배운 것** — 사용자의 진짜 동기를 surface 하기 전에 "이건 over-engineering 아닐까" 같은 일반론 코칭 금지. 동기를 묻고, 그 다음에 함께 판단하는 게 맞다.

## 8. 자정 넘어, D01 을 미루기로 결정

00:33. 모든 게 준비됐다. 다음 한 줄이면 D01 implementer subagent 가 dispatch 된다 — fleet-state private repo 가 GitHub 에 생기고, clone 되고, 첫 commit 이 push 된다.

근데 멈췄다.

이유는 단순했다. fleet-director 의 본래 목적이 "강대종 hands-off 시간 확보" 인데, 자정 넘어 시작 = 본말전도. D01 산출물 (fleet-state repo 의 첫 형태와 운영 관습) 은 D02~D14 의 토대라 review 품질이 중요. 그건 자정 직전에 굴리는 게 아니다.

대신 트리거 메모리를 박았다.

```
~/.claude/projects/-/memory/fleet_director_d01_trigger.md
새 세션 첫 메시지 "D01 시작" → plan 의 Day 1 implementer subagent 즉시 dispatch
```

내일 컨디션 좋을 때 챗에 한 줄 "D01 시작" 이면 자동 시작. D02~D14 도 같은 패턴.

> **배운 것** — 자정 직전 만든 plan 은 자정 직전 시작하지 마라. 만드는 시간 ≠ 시작하는 시간. 시작하는 시점은 review 품질이 잘 나오는 시간.

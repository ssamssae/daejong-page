## 3. reviewer 가 PASS 라고 한 SIGPIPE 가 사실 active blocker 였다

spec-reviewer subagent dispatch. mac-mini SSH 해서 양 기기 parity 검증 + 6항목 inspection. 결과는 깔끔했다: bc2bbbf 6/6 PASS, Mac 본진 10/10, mac-mini 7/10 (3 fail = push-race env, 코드 무관). 정확히 우리가 방금 등록한 test harness push 격리 follow-up 의 현상.

code-quality-reviewer subagent dispatch. 결과: BLOCKING 0건, NON-BLOCKING 7건. 그중 1번이 이렇게 적혀 있었다.

> `git diff --cached -G '...' --name-only | head -1 | grep -q .` is the same `pipe | head | grep -q` shape as the D05#6 family. In practice it's safe here because the pipeline lives inside `if ...; then`, which suppresses `set -e` on the pipeline's exit (even when pipefail catches SIGPIPE=141).

reviewer 가 "set -e 가 if 안에서 suppress 되니까 safe" 라고 했다. 맞다, set -e 는 if 안에서 suppress 된다. 하지만 그건 abort 막는 얘기지, 조건이 true/false 로 평가되는 거랑은 다르다. `pipefail` 에서 upstream SIGPIPE 가 141 로 surface 되면, `if [pipeline=141]` 은 false 분기로 빠진다. grep 이 match 했어도.

empirical repro 한 줄.

```bash
$ bash -c 'set -euo pipefail; r=0; if seq 1 1000000 | head -1 | grep -q .; then r=1; fi; echo $r'
0
```

기대값 1, 실제값 0. 버그 재현. 그래서 reviewer 가 non-blocking 으로 마크한 그 한 줄이 사실 active blocker 였다. memory 의 same-family 3rd-instance 룰: 같은 failure family (SIGPIPE under pipefail) 가 3번째로 발견되면 functional impact 0회여도 active blocker. D05#6 에서 3건 close 됐는데 bc2bbbf 가 2건 새로 추가 = 4·5번째 인스턴스. 다음 Day 시작 전 즉시 패치.

`31a3635` 한 commit 으로 두 군데 fix. post-action-scan 의 huge-diff 가지는 `first_match=$(... | head -1) || true` 후 `[ -n "$first_match" ]` 검사 (capture-then-test). stale-recover 의 entry-point dirt 검사는 `_porcelain=$(git status --porcelain) || true` 후 `grep -qE '...' <<<"$_porcelain"` (here-string). `|| true` 가 pipefail 의 141 을 흡수하고, here-string 은 그냥 pipe 가 아니라서 SIGPIPE 발생 자체가 불가능하다.

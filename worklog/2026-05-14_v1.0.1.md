> 11:22 KST, 강대종이 "뭐할까" 한 마디를 던졌다.
> 그 직후 데스크탑3060Ti 가 mac mini 의 4시간 침묵을 들고 왔다.
> 13:26, cache 한 줄 + 17줄짜리 룰로 봇이 다시 살았다.

새벽 사이클(v1.0.0) 이 deep-work SoT 신설로 끝났을 때, 다음 사이클이 6시간 뒤에 시작될 줄 몰랐다. 점심 직전 강대종이 본진 챗에 "뭐할까" 한 마디를 보낸 게 11:22 KST. 그 메시지가 발사되자마자, 다른 노드(데스크탑3060Ti) 가 도움 요청을 들고 왔다 — mac mini 텔레그램 봇이 4시간째 입을 다물고 있었다.

## 1. mac mini 가 4시간 동안 입을 다물고 있었다

데스크탑3060Ti `mac-report` 가 도착했을 때 본진은 이미 새벽 사이클 분배를 짜고 있었다. 그래서 한 채팅에 두 트랙이 겹쳤다: (1) parkinglot 잡일 양 트랙 분배, (2) mac mini @MyClaude4 봇 침묵 디버깅.

침묵의 단서가 묘했다 — claude PID 는 살아있고, `settings.json` `enabledPlugins.telegram` 도 `true` 였다. 모든 게 정상으로 보이는데 ToolSearch 에서 `telegram` 을 검색해도 0건. 그리고 `/plugin` 슬래시를 치니 명시적 메시지가 떴다:

```
Failed to reconnect to plugin:telegram:telegram
```

이게 결정적이었다. plugin **cache 가 손상됐다**. 데스크탑3060Ti 가 SSH 로 들어가서 옵션 A(`settings.json` 토글 → in-flight 무효), 옵션 B(tmux kill + `launchctl kickstart` → MCP server spawn 여전 X) 까지 시도하고 막힌 후, 강대종이 Termius 콘솔에서 옵션 C 를 직접 실행:

```bash
rm -rf ~/.claude/plugins/cache/claude-plugins-official/telegram
# /exit → claude --dangerously-skip-permissions
```

`/mcp` 화면에 `plugin:telegram:telegram ✓ connected · 4 tools` 가 떴다. 11:23 KST. 강대종이 "되나", "되냐지" 하니 봇이 typing indicator 로 답했다.

> **배운 것** — plugin 가 silent fail 할 때 `/plugin` 슬래시의 reconnect 에러 메시지가 단서. claude 자식 프로세스 0건 확인이 가장 빠른 진단.

## 2. 데스크탑3060Ti 가 본진 대신 콘솔에 들어갔다

이번 사고의 묘미는 mac mini 가 자기 자신을 못 고친다는 점이었다. mac mini 의 챗봇이 죽었으니 본진→mac mini 디렉티브가 안 닿는다. 그래서 본진이 데스크탑3060Ti 에 SSH 가이드 위임 → 데스크탑3060Ti 가 mac mini 콘솔에서 진단 → 강대종이 Termius 로 한 줄씩 실행.

본진은 그 사이 다른 노드들도 같은 사고가 잠재해있나 점검했다 — 본진/WSL/desktop3060ti/hermes 4 노드 모두 `0.0.6/` 디렉토리 정상 + claude 자식 프로세스 존재. **단발 사고 확정**.

> **배운 것** — 한 노드가 죽으면 다른 노드가 SSH 로 들어가는 게 깔끔하다. 본진이 모든 걸 직접 디버깅할 필요 없다.

## 3. 두 트랙 10 iter — parkinglot 이 자동으로 6+4 로 떨어진 이유

병행으로 진행한 양 트랙 dynamic /loop. 이게 두 번째 사이클(새벽 9 iter 후) 이다. 본진이 parkinglot 22 건을 분배할 때 기준 하나만 적용했다 — **"1 iter 안 완결 가능한 잡일만"**. WebSearch 1번, 단일 파일 변경, draft 작성 수준.

- Mac 트랙 6 (rtk·secall·STT·메모요 ASO·평가버튼·loop-context plan)
- WSL 트랙 4 (dead-ends detail.html · 3 BACKLOG · Codex 진단 · ASC Playwright 방향)
- 큰 작업 = 별도 백로그, mac mini/3060Ti 작업 = 위임 트랙으로 분리

남은 게 정확히 6+4. 우연이 아니라 사전 필터 기준이 분명해서 자연히 떨어진 거다.

dynamic /loop 자체 픽 순서도 우연이 아니었다. **가벼운 것부터 점점 무거운 것으로** 정렬됐다. secall(WebSearch 1) → rtk(검증만) → STT(WebSearch + 비교) → ASO(텍스트 작성) → 평가버튼(코드 draft) → loop-context(plan 작성). 마지막은 가장 무거워서 plan 까지만, 실제 구현은 별 작업으로 분리.

> **배운 것** — dynamic 의 self-pace 가 좋은 건 모델이 매 iter 가장 쉬운 것부터 픽한다는 점. 사람이 미리 순서 짜지 않아도 자연히 가벼움 → 무거움 순서로 흐름.

## 4. RC1 — 봇들이 서로 헷갈리지 않게 17줄로 막은 룰

WSL Task C 가 잡아낸 문제는 5/8 부터 잠재해있던 사고였다. mac mini Codex 시대에 `codex-to-wsl.sh` 가 (a) tmux paste 풀 directive + (b) Telegram `[🏭→🪟] [명령]` 80자 요약을 두 채널로 보냈는데, WSL chatbot 이 (b) 80자 요약을 새 명령으로 또 처리. **같은 명령 2회 수신**.

진단 결과 root cause R1 = WSL chatbot CLAUDE.md 에 "protocol header skip 룰" 미적용. AGENTS.md(OpenClaw) 에는 있었지만 WSL chatbot 측 컨텍스트엔 없었음.

처방 = `globals/CLAUDE.md` 에 17줄 추가:

```
정규식 ^\[.+→.+\]\s+\[.+\]\s+.+$ 매칭 메시지 = 단순 시각적 알림
chatbot 행동 / 새 turn 시작 / 작업 실행 안 함
```

PR #37 13:26 KST 머지. SoT symlink 라 본진/WSL/mac-mini 양 기기 자동 sync. mac mini 가 오늘 Codex → Claude Code 로 전환됐으니 Codex 자체는 폐기됐지만, 룰은 모든 cross-device 알림에 여전히 유효.

> **배운 것** — chatbot 행동 룰은 매 turn 자동 로드되는 CLAUDE.md 자리. AGENT.md(on-demand) 는 절차/워크플로우 자리. RC1 같은 게 헷갈리면 CLAUDE.md 가 점점 부풀어 나중에 토큰 비용 폭증.

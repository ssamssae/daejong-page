## 1. 패치가 통했는지 확인하러 들어갔는데, 다른 문제가 있었다

오전 9시 사건은 plist 의 `EnvironmentVariables.PATH` 가 빠져서 `bun` 을 못 찾던 거였다. 17:44 KST 에 PATH 박고 launchctl reload. 검증은 "다음 재부팅 또는 tmux kill + launchd respawn 까지 대기" 로 deferred.

지금 SSH 로 들어가 보니 — `bun PID 1560` 이 정상 polling 중이었다. PATH 패치는 살아남았다. **MCP server spawn 실패는 재현되지 않았다.** 그러면 강대종이 본 "안 됨" 은 뭐였을까.

`tmux capture-pane` 으로 mac mini 의 claude 화면을 직접 읽어보니, pane 에 다음 줄이 있었다:

```
Listening for channel messages from: plugin:telegram@claude-plugins-official
```

이 줄이 보인다 = 현재 세션은 `--channels` 플래그가 활성. 그런데 강대종 보고는 "처음엔 안 되고 두 번째 cc 진입 때 됨". 즉 **첫 진입은 다른 세션을 보고 있었다**는 의미.

## 2. 두 명령이 한 글자 차이로 갈렸다

`~/Library/LaunchAgents/com.user.tmux-claude.plist` 의 부팅 명령:

```bash
tmux has-session -t claude 2>/dev/null \
  || tmux new -d -s claude /opt/homebrew/bin/claude --dangerously-skip-permissions
```

`~/bin/cc` 의 CLAUDE_CMD:

```bash
claude --channels plugin:telegram@claude-plugins-official
```

`--channels` 가 한쪽에만 있다. launchd 가 부팅 시 만든 `claude` 세션은 outgoing 툴만 동작했고, incoming polling 은 비활성이었다. 강대종이 `m` → `cc` 로 들어가면 cc 가 `tmux has-session -t claude` 체크 → YES → 기존 세션 재사용. "연결안된상태" 처럼 보인다. /exit + cc --new 하면 cc 가 fresh 세션을 자기 명령으로 띄워서 비로소 incoming listen 활성.

> **배운 것** — 같은 도구를 두 경로(launchd RunAtLoad / 셸 래퍼)로 띄울 땐, 그 도구의 명령행 전체가 한 글자도 어긋나지 않아야 한다. 부분 일치는 "거의 됨" 처럼 위장되는 좀비 상태를 만든다.

플래그 추가 + launchctl bootout/bootstrap reload. 23:34 KST 패치 마침. 현재 alive 세션은 건드리지 않았고, 다음 재부팅이나 cc --new 때 진짜 검증.

Mac mini claude 가 본인을 Mac 본진이라고 답하고 있었다. 자기 챗에서 자기를 부정.

`hostname` 확인 = `mac-mini`. CLAUDE.md 매핑 표 = `@ssamssae_claw_bot` → 🏭 명시. 둘 다 맞는데 왜 claude 가 🍎 를 골랐을까.

(2)(3) 둘 다 LLM 가 능동적으로 `getMe`/`hostname` 쉘 호출해야 가능 — **SessionStart context 에 자동 박히지 않는다.** 정체 단서 0개 → LLM 가 priori 로 추론 → CLAUDE.md 의 첫 매핑 row 가 Mac 본진 (🍎) → 그게 기본값.

별 검증: Mac 본진 `.env` 도 동일하게 `TELEGRAM_BOT_USERNAME` 미설정. 단 본진은 hostname (`USERui-MacBookPro.local`) 이 CLAUDE.md fallback 규칙에 직접 매칭돼 운 좋게 통과. **5 노드 전부 같은 사각지대를 안고 있었던 셈.**

## 4. 작은 패치를 두 번 만에 통과시킨 또 다른 함정

`.env` 에 `TELEGRAM_BOT_USERNAME=ssamssae_claw_bot` 한 줄 추가. cc --new 로 세션 재기동. 1차 재기동 — `pgrep -lf "bun.*telegram"` 결과 0. claude 는 떴는데 bun 이 안 떴다.

원인은 어이없게도 오전 그 사고와 동일. 내가 SSH 로 cc --new 를 돌릴 때 export 한 PATH 가 `/opt/homebrew/bin` 까지만이라 `~/.bun/bin` 이 빠져있었다. plist 는 `EnvironmentVariables.PATH` 박혀 있어 부팅 시엔 괜찮지만, 내 SSH 세션이 launchd 가 아니라 그 영향을 못 받았다.

```bash
ssh mac-mini 'export PATH=/Users/user/.bun/bin:/opt/homebrew/bin:$PATH; /Users/user/bin/cc --new'
```

2차 재기동에서 PATH 박고 통과. `claude PID 5160 --channels` + `bun PID 5196 plugin:telegram MCP` 둘 다 검증.

> **배운 것** — 한 번 잡았던 PATH 함정은 동일한 사용자가 동일한 워크플로우 안에서 한 시간 안에 또 밟을 수 있다. 자동화의 PATH 가 사람의 PATH 가 아니라는 사실은 사람의 직관과 영원히 충돌한다.

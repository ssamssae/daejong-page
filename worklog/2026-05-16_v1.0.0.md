## 1. parking-lot 의 한 줄이 사이클을 열다

`bc64643` 커밋이 01:29 KST 에 박힌 직후, 본진은 별 ack 없이 진단 사이클을 자율 진입했다. RAM 업그레이드는 비가역 외부 영향 (돈)이지만, **측정 자체는 read-only** 라 그냥 돌렸다. `vm_stat`, `sysctl vm.swapusage`, `memory_pressure`, `ps aux | sort -k 6 -rn | head -10` — 네 명령이 30초 안에 답을 줬다.

```
swap used:           318 MB / 2 GB    → < 1 GB 임계
memory_pressure:     normal (85% free)
Top RSS:             WindowServer 1.36GB / claude 658MB
                     Chrome 합 ~570MB / Ghostty 123MB
활성 컨텍스트:        tmux 7 / Chrome Helper 5 / Xcode 0
                     Simulator 백그라운드 서비스만
```

16GB 못 채우고 있었다. 다 합쳐도 3GB 미만. parking-lot 의 의문은 "체감만 있고 데이터는 없음" 으로 마감.

## 2. 다섯 기기를 모아보니 96GB 였다

본진 데이터만으로는 좁다. 🖥 desktop3060ti 가 5대 기기 RAM 조사를 routing 받았다. mac-report.sh 3-channel 패턴 (1차 paste / 1.5차 reverse reply / 2차 강대종 Telegram) 으로 한 사이클 돌아왔다.

| 기기 | 이모지 | 물리 RAM | WSL2 할당 |
|---|---|---|---|
| Mac 본진 (MBP) | 🍎 | 16 GB | — |
| Mac mini | 🏭 | 16 GB | — |
| WSL (Ryzen 3900X) | 🪟 | 32 GB | 15 GB |
| 데스크탑 3060Ti | 🖥 | 16 GB | 7.7 GB |
| 노트북 3060 (Legion 5) | 💻 | 16 GB | 7.7 GB |

WSL 본진만 32GB. 🖥/💻 는 WSL2 default 8GB 라 LLM/SD/Whisper 돌릴 때 헤드룸 빠듯. **합 96GB 의 분포가 이 5노드 운영 구조의 실제 모양** 이었다.

## 3. .wslconfig 가 없다 = 7.7GB default 였다

🖥 후속 진단에서 `.wslconfig` 파일이 아예 없는 게 확인됐다. 즉 Windows 11 default 정책 (host RAM 의 50% 또는 8GB 중 작은 값) 그대로 동작 중. 7.7GB 관측치와 정확히 일치.

세 옵션을 표로 던졌다.

| 옵션 | memory | swap | Windows 남음 | 적합 케이스 |
|---|---|---|---|---|
| α | 12 GB | 8 GB | 4 GB | LLM/SD 자주 + Steam 거의 안 함 |
| β | 10 GB | 4 GB | 6 GB | 게이밍 가끔 / 안전마진 |
| γ | 14 GB | 4 GB | 2 GB | Windows 압박 — 비추 |

추가 권장 (어느 옵션이든):

```ini
[experimental]
autoMemoryReclaim=gradual
sparseVhd=true
```

전자는 WSL2 가 idle 일 때 메모리 점진 반환, 후자는 VHD 파일 자동 축소. 부수효과 = `wsl --shutdown` 필요 → 🖥 tmux 'claude-main' 끊김 → 본진 reverse reply 경로 일시 단절. 그래서 강대종 한 글자 ack (α/β) 받기 전엔 hold.

> **배운 것** — 옵션 비교는 표 + 한 글자 ack 패턴이 의사결정 마찰을 가장 줄인다. 본문 산문체보다 표 한 장이 더 빠른 결정을 만든다.

## 4. "WSL 이 더 빠른 것 같다" — 그건 사실 API 였다

01:44 KST, 강대종이 다른 각도로 들어왔다. "체감상 wsl이 클로드코드 응답이 더 빠른 것 같아서 3900X 이기도 하고 32G 라서그런건지 GPU 도 2070S 라서 더 그런 것 같기도 하고... wsl로 변경할까 고민".

여기서 본진은 한 가지 사실을 다시 surface 해야 했다. **Claude Code 응답 속도는 사실상 100% Anthropic API 측 latency 가 지배한다.** 로컬 CPU/RAM/GPU 영향 거의 0. 2070S 가 응답을 빠르게 만드는 게 아니다.

체감 차이의 진짜 원인 후보 (가능성 큰 순):

1. **네트워크 회선** — 본진 wifi / WSL 유선? `ping -c 10 api.anthropic.com` 한 번이면 보임
2. **터미널 렌더링** — Ghostty 가 본진에서 무거운지? native Terminal 비교
3. **백그라운드 throttling** — App Nap / pmset (직전 `mac_throttling_diagnosis` 메모리)
4. **WindowServer 1.36GB** — 외장 모니터 / 창 압박

본진 이전은 단순 기기 교체가 아니다. MACHINE_ROLES.md / 텔레그램 봇 라우팅 / mac-report.sh 류 스크립트 / cross-device directive — 전부 본진 중심으로 lock 된 구조다. WSL 이 본진 역할 받으려면 가동 패턴 (낮 ON / 밤 OFF → 24/7) 부터 재설계.

세 옵션을 던졌다 — (i) 본진 유지 + 진단 (**추천**), (ii) WSL 본진 이전 (진단 후 H/W 확정되면), (iii) 하이브리드 (현행, 이미 이 모드). 강대종이 "자연어로" 한 마디 던지고 옵션 본문이 너무 잘게 쪼개진 걸 환기. 다음 답변부터 산문체로.

> **배운 것** — 체감을 결정 근거로 쓰면 잘못된 H/W 투자로 이어진다. "WSL 이 빠르다" 같은 체감은 네트워크/터미널/throttling 어디서나 만들어질 수 있다. 본진 이전 같은 비가역 결정은 진단 측정 마치고서.

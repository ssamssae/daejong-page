작업일지 v1.0.0 이 새벽 RAM 진단으로 끝난 직후의 자연스러운 연결편. v1.0.1 의 이야기는 두 갈래로 시작해서 한 갈래로 합쳐진다. 한 갈래는 mesh-vote / trio-vote 의 자율 사이클 5 PR 머지였고, 다른 갈래는 새벽부터 미뤄둔 D07 launchd install 의 실호출이었다. 두 갈래가 합쳐지는 지점에서 mac-mini 가 5초짜리 자동 발자국 4개로 pull 을 거부했고, 그걸 풀고 나니 worker 들이 깨어났다.

## 1. Ghostty 가 잠들고 있었나

새벽 v1.0.0 의 마지막 의문 — "본진이 멈춰 보이는데 ping 차이는 1.8ms 뿐" — 의 후속 진단을 11:00 KST 부터 시작. caffeinate 는 살아있고 (Claude Code 가 자체 호출하는 `-i -t 300`), pmset 의 sleep / displaysleep / disksleep 다 0. OS 수준 throttling 은 깨끗했다. 그러면 남는 가설은 Ghostty 의 App Nap 뿐이었다.

Ghostty 의 공식 config reference 를 정독했는데 App Nap / NSAppSleepDisabled / background throttle 옵션이 0건. issue tracker 에서 `NSAppSleepDisabled` 검색 0건, `App Nap` 검색에서 #12071 한 건이 closed 로 떠있었는데 환경이 거의 동일했다 — M4 Pro / macOS 26.4 / Ghostty 1.3.1 / `command = direct:claude` 의 idle silent exit. 형님 현상은 stuck (process alive, 응답 멈춤) 이라 silent exit (process 죽음) 와는 좀 달랐지만 같은 root cause 일 가능성. trio-vote 만장일치 (iii) 사전 조사 결정 후 `defaults write com.mitchellh.ghostty NSAppSleepDisabled -bool true` 가 standard 대응으로 확정. 적용 + `defaults read` = 1 검증. Ghostty 재시작 + 며칠 stuck 빈도 A/B 관측은 형님 손/시간 트리거 별 follow-up.

> **배운 것** — 체감 ("느려요") 이 데이터 ("ping 차이 1.8ms") 보다 강할 때, OS 수준 측정 + 도구 issue tracker 정독 둘 다 거쳐야 진단이 닫힌다.

## 4. PR 스택의 함정

mesh-vote 잔존 (2) 판정 키워드 다국어 PR #52 + 잔존 (3) Phase 3 race condition fix PR #53 가 stack 으로 박힌 채 형님이 "한번에 머지" 트리거를 줬다. #52 가 main 으로 squash 머지 되자마자 (8872b8a) #53 의 mergeStateStatus 가 DIRTY / CONFLICTING 으로 떨어졌다. 이유는 단순했다 — squash 머지가 새 commit hash 를 main 에 박았는데, #53 branch 는 v0.3.2 branch 위에 stack 돼있어서 old v0.3.2 auto-commit 들을 base 로 갖고 있었기 때문이다.

해결은 git rebase 한 번이었다. `git fetch origin main && git rebase origin/main` 이 upstream 으로 흡수된 patch 2건을 자동으로 drop 하고 남은 race condition fix 3 commit 만 위로 끌어올렸다. force push 후 sleep 5 초 기다리니 mergeStateStatus 가 CLEAN / MERGEABLE 로 돌아왔다. squash 머지 (6b9b5ea). mesh-vote v0.3 stack 4 PR (v0.3 + v0.3.1 + v0.3.2 + v0.3.3) + trio-vote v0.2 PR — 합쳐서 5 PR 다 main 박힘.

> **배운 것** — PR stack 에서 base PR 이 squash 머지 되면 stack 위의 PR 은 무조건 rebase 한 번을 거친다. squash 가 새 commit hash 를 박기 때문.

## 6. mac-mini 의 5초짜리 자동 발자국 4개

D07 install 의 첫 step 인 mac-mini 진입 직전에 mac-mini 의 agent-fleet-state 가 `git pull --ff-only` 를 거부했다. local 에만 있는 commit 4개와 origin/main 에만 있는 commit 9개가 동시에 박혀있었다. local 4개를 정밀 인스펙트한 결과 정확히 5/13 19:46:47~52 의 5초 동안 fleet primitive 가 KILL human_emergency → kill --release → mission propose → cleanup 자동 흐름을 끝낸 발자국이었다. PAUSED 파일 +6 → -6, mission.json + decompose +29 → -29, working tree net 변경 정확히 0.

형님이 "정확히 판단하고 진행해야지 C" 라고 정밀 인스펙트를 요구했고, 그 검증을 거친 뒤에야 hard reset 으로 갔다. blanket reset 보다 commit 별 내용 확인이 우선이라는 룰의 forcing function. `git reset --hard origin/main` 한 줄로 HEAD = e406f7b, diff clean.

> **배운 것** — 자동 발자국으로 보이는 commit 도 inspection 없이 hard reset 으로 폐기하지 말 것. 5초짜리 self-canceling 도 형식상 commit 이라 영구 삭제가 가역 아님.

## 7. worker 들이 깨어났다

| 노드 | 메커니즘 | 주기 | label / 라인 |
|---|---|---|---|
| 🏭 mac-mini | launchd | 2분 | `com.fleetdirector.macmini` |
| 🍎 본진 | launchd | 5분 | `com.fleetdirector.macbook` |
| 🪟 WSL | cron | 5분 | `*/5 * * * * /bin/bash .../run-worker.sh` |

mac-mini install 후 강제 fire 했을 때 log/launchd-macmini.err 에 "INFO: stale-recover cleaning leftover recover-scope dirt" 라인이 박혔다. 본진 install 후 강제 fire 했을 때도 같은 라인. worker 가 첫 깨어나서 이전 stale state 를 정리한 정상 흔적이었다. WSL 은 갓 clone 한 깨끗한 노드라 stale-recover 라인이 안 박혔고 silent exit — 큐가 비어있어서 깔끔히 빠져나온 자연 동작.

이로써 D07 launchd install 흐름이 완전히 닫혔다. fleet-director 시스템이 본격적으로 task queue 의 잡을 device_pref 따라 자동 분배 시작. desktop3060ti / notebook3060 은 D07 spec 자체에 worker plist 가 없어서 별 사이클이 필요하다.

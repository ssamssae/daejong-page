## 2. "작업 없다며"

11시 35분, 형님이 텔레그램으로 한 줄을 던졌다.

> "그리고 메모요 세션은 이제 종료처리해도 되지 않나? 작업 없다며"

진단이 정확했다. cron 이 #4 부터 #10 까지 굴린 건 전부 사전 작업이었고, 그 사전 폭도 거의 다 떨어진 상태. 다음 fire 가 들어와도 무한 marginal polish 만 깎고 있을 게 뻔했다. 깔끔하게 끄는 게 맞았다.

조치는 세 줄이었다 —
```bash
launchctl bootout gui/$(id -u) ~/Library/LaunchAgents/com.daejongkang.memoyo-sync-loop.plist
tmux kill-session -t memoyo-loop
git push origin mac/memoyo-sync-2026-05-17   # SYNC_PROGRESS.md 종료 마커
```

`launchctl bootout` 명령이 한 번 I/O error 를 돌려줬지만 verify 보니 이미 unload 된 상태. plist 파일 자체는 `~/Library/LaunchAgents/` 에 보존했다 — Firebase 프로젝트 만들면 `launchctl bootstrap` 한 줄로 그대로 복귀 가능.

> **배운 것** — 자율 루프가 비가역 hold 에 막히면 그 시점에 끄는 게 정답. "한 번 더 fire 하면 뭐라도 나오겠지" 같은 미신은 무한 marginal polish 만 깎는다.

## 5. Mac mini 가 들고 있는 터널

다음 결정 — 누가 cloudflared 데몬을 호스팅할지. 형님 브라우저는 Mac mini 위에서 돌고 있었고, 처음엔 "Mac 본진 = 큐 파일 있는 곳, 본진 설치가 단순" 으로 추천했다. 형님이 30초 후 되물었다 —

> "꼭 본진에서 하는 걸 추천해?"

이 한 마디로 추천이 뒤집혔다. 솔직히 보면 — Mac mini 가 진짜 24/7 server-grade(launchd 워커들 + 디스플레이 X + 일부러 서버 용도로 설계됨), 본진은 "사실상 24/7" 이긴 해도 사람이 hands-on 으로 쓰는 작업 머신. 본진 sleep / restart / 업그레이드 할 때 큐 대시보드 끊김. 서비스는 서버에, 작업은 작업 머신에 — 그 분리 원칙이 맞았다.

Mac mini 에 cloudflared 설치는 본진에서 SSH 로 `brew install cloudflared`(arm64, v2026.5.0) → 형님이 Mac mini Terminal.app 열어서 `sudo cloudflared service install <token>` 한 줄. Mac mini sudo 가 passwordless 아니라 형님이 직접 비밀번호 한 번 쳤다.

```
2026-05-17T04:13:44Z INF Installing cloudflared client as a system launch daemon. cloudflared client will run at boot
2026-05-17T04:13:44Z INF MacOS service for cloudflared installed successfully
```

본진에서 verify — `pgrep cloudflared` 2 PIDs(메인 18792, 자식 18831), 3 connections 등록(icn06/icn01 PoP, QUIC 프로토콜). Tunnel **HEALTHY** ✓.

> **배운 것** — 호스팅 결정은 "지금 어디 코드 있는지" 가 아니라 "이 서비스가 1년 후 어디서 돌아야 하는지" 로 한다. 본진은 작업 머신, Mac mini 는 서버 — 그 경계가 흐려지면 작업 머신 sleep 으로 서비스가 끊긴다.

## 6. 인증의 작은 게이트

Public application route 한 줄 추가(Subdomain=`queue`, Domain=`kangdaejong.com`, Type=HTTP, URL=`localhost:7777`) 로 라우팅이 살아났다. `dig queue.kangdaejong.com` → CF edge IP, `curl https://queue.kangdaejong.com` → HTTPS 502 Bad Gateway. tunnel → cloudflared(Mac mini) → localhost:7777 경로는 살아있고, 마지막 잎(FastAPI) 만 비어있는 상태.

이제 누구나 그 URL 을 알면 들어올 수 있다. 형님 todos.md 가 인터넷에 노출되는 건 곤란하니 인증 한 겹.

Cloudflare Access — Self-hosted application, policy `Allow Daejong`(Selector=Emails, Value=`ssamssae@naver.com`), Login methods One-time PIN 만 켜고 다른 OAuth 다 끔. Session 24h. MFA / Purpose justification / Temporary auth 다 OFF. Browser-RDP 도 OFF. 형님 한 명만 통과시키면 되는 게이트라 over-engineering 피했다.

활성화 후 같은 URL `https://queue.kangdaejong.com` 다시 치니까 이번엔 302 redirect 가 떨어진다 — `late-bird-2983.cloudflareaccess.com/cdn-cgi/access/login/...`. CF Access 로그인 페이지로 자동 리다이렉트. 들어가면 이메일 입력 → OTP 발송 → 통과 → (FastAPI 안 떠서 또 502).

게이트 작동 완료. 14시 15분.

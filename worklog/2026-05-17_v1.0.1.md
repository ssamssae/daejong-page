## 2. 작업일지 탭에 박힌 빨간 "주의요망"

같은 회사 페이지에서 "작업 일지" 카드를 클릭했더니 work.kangdaejong.com 으로 이동했고, Chrome 이 URL 바에 빨간 "주의 요망" 배지를 띄웠다. "이 사이트는 보안 연결(HTTPS)이 사용되지 않았습니다."

진단하니 서버가 `*.github.io` wildcard cert 를 그대로 내주고 있었다. work.kangdaejong.com 용 cert 가 아예 발급 안 된 상태. `gh api repos/.../pages/health` 로 확인했더니 `is_https_eligible: true` 인데 `https_error: peer_failed_verification` — Let's Encrypt 발급 자격은 있는데 큐에서 5시간째 멈춰있던 것.

> **배운 것** — GitHub Pages cert provisioning 은 "자동" 이라 광고하지만 큐에서 영구히 멈출 수 있다. 30분 timer + 진단 + 강제 재트리거 트릭을 알아두지 않으면 운영 중에 사용자한테 빨간 경고가 그대로 노출된다.

해결법: CNAME 을 잠시 비웠다 다시 박는 토글 트릭.

```bash
gh api -X PUT repos/ssamssae/daejong-page/pages -F 'cname='
sleep 8
gh api -X PUT repos/ssamssae/daejong-page/pages -F 'cname=work.kangdaejong.com'
```

30초 후 `authorization_pending` → `approved`. expires_at 2026-08-15. `https_enforced=true` 활성화. cert subject 확인하니 `CN=work.kangdaejong.com` — 정상.

이 사고 자체를 이슈로 박았고 (`2026-05-17-gh-pages-cert-provisioning-stuck.md`), 노하우로도 박았다 (`2026-05-17-gh-pages-cert-cname-toggle.md`). 다음에 같은 함정을 또 만나도 30초 안에 끝낼 수 있게.

## 3. 5노드 mesh 가 자기 신원을 안 박은 사고

밤 늦은 시점에 데스크탑3060Ti 노드가 daejong-page habits/issues 편집 결과를 mac-report.sh 로 본진에 보냈다. 그런데 본문 첫 줄이 "본 노드가 처리한..." 으로 시작해서 어느 노드 보고인지 식별이 안 됐다. 형님 폰에는 `[🖥→🍎] [결과] ...` 알림이 떨어졌는데 본진 챗봇은 봇끼리 수신 차단 룰 때문에 그 prefix 를 못 봤다.

형님 질문: "왜 출처불명 노드로 나오지?"

진단하니 — 운반체 두 채널 (텔레그램 알림 + tmux paste) 이 따로 노는데 본진 챗봇은 tmux paste 본문에만 의존한다. 그 본문에 sender 신원이 안 박혀있었던 게 직접 원인. 맥미니/WSL 챗봇은 첫 줄에 자기 신원을 박는 컨벤션을 따랐지만 데스크탑만 빠뜨려서 발생.

trio-vote 호출. PM/엔지니어 모두 [B] (스크립트 자체에 자동 prepend), 비판론자 [D] (이중 안전벨트). 다수결 [B] + 형님 직접 [B] ack.

`mac-report.sh` 에 `detect_sender_id()` 함수 추가. hostname 기반 추론 + FROM_DEVICE arg override + Idempotent guard (body 첫 줄에 이미 sender prefix 박혀있으면 skip — 비판론자 우려 직접 반영). 5노드 git pull propagate.

> **배운 것** — 5노드 chatbot 컨벤션 누락에 의존하지 말고 스크립트 자체를 단일 진실원으로 만들어라. forcing function = chatbot 손이 떨릴 가능성을 코드로 차단.

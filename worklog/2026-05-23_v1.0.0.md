# 2026.05.23 작업일지 v1.0.0 — 챗봇이 켜져 있는 줄만 알았다

> 23:18, 형님이 노트북을 재부팅했더니 챗봇이 안 붙었다.
> 00:14, 라이젠은 "Listening"이라고 적힌 채 메시지를 한 통도 못 받고 있었다.
> 00:43, 3노드 모두 진짜 부팅 사이클에서 사람 손 0으로 살아났다.

부팅하면 챗봇이 알아서 깨어나야 하는데 매번 누군가가 Enter 한 번을 박아줘야 살아나던 시간이 있었다. "Listening for channel messages" 라는 출력 한 줄만 보고 다 됐다고 믿어버린 건 더 오래된 false positive 였다. 오늘은 그 두 층을 다 벗겨낸 사이클이다.

## 1. 노트북 재부팅 후, 챗봇이 자기 자신을 기다리고 있었다

형님이 노트북을 수동으로 재부팅하고 "ㅎㅇ"을 네 번 보냈는데 답이 0이었다. SSH로 들어가보니 `tmux 'claude' session`도 살아있고 `claude` 프로세스도 도는 중이었다. 다만 화면이 한 장면에서 멈춰 있었다.

```
❯ 1. Yes, I trust this folder
  2. No, exit
```

Claude Code 가 Linux/Ubuntu 위에서 비대화형 systemd로 켜질 때마다 띄우는 워크스페이스 trust 질문. stdin이 TTY가 아니라 답을 받을 방법이 없었다. macOS launchd에선 안 뜨는, 정확히 우분투 3노드(라이젠/데스크탑/노트북) 한정 게이트였다.

## 2. 두 가설이 차례로 부서졌다

- **가설 A** — `--dangerously-skip-permissions` 플래그가 trust도 우회할 것이다. 맥미니 launchd가 그 플래그로 잘 돌아가니까. → 박았는데 trust 화면 그대로.
- **가설 B** — 사람이 한 번 1을 박으면 어딘가에 저장될 것이다. macOS는 안 뜨니까. → restart 한 번에 다시 트러스트 화면. Linux는 매번 새로 묻는다.

세 번째 시도가 답이었다. systemd unit의 `ExecStartPost`에 한 줄을 끼웠다.

```
ExecStartPost=/bin/bash -c 'sleep 6 && tmux send-keys -t claude Enter'
```

trust 화면의 디폴트 ❯가 이미 "1. Yes"에 박혀있어서, 그냥 Enter 한 번을 자동으로 박아주면 통과한다. sed 한 줄로 노트북·라이젠·데스크탑 unit에 동일 패턴을 깔았다. 노트북 자동 재부팅 검증 = 발사 23:41:44 → SSH 복귀 23:42:43 → polling 시작 23:42:55. **1분 11초만에 사람 손 0으로 풀복구**.

> **배운 것** — Linux 의 trust prompt는 영구 저장되지 않는다. `--dangerously-skip-permissions`는 permission 우회일 뿐 trust 우회가 아니다. 게이트는 게이트로 풀어야 한다.

## 3. 라이젠은 "Listening" 이라고 적힌 채 거짓말을 하고 있었다

데스크탑은 같은 fix로 폴링이 살아났는데 라이젠만 안 됐다. `tmux capture-pane` 으로 보면 "Listening for channel messages from: plugin:telegram@claude-plugins-official" 가 또렷이 박혀있는데, `getWebhookInfo` 의 `pending_update_count` 는 6에서 8로, 8에서 9로 계속 늘어났다. 메시지가 큐에 쌓이는데 봇이 안 받는 상태. **출력 한 줄이 polling 시작을 보증하지 않는다는 false positive**.

claude 안에서 `/mcp` 메뉴를 열어보니 진단이 한 줄로 잡혔다.

```
plugin:telegram:telegram · ✘ failed
```

`reconnect` 시도하니 `ENOENT`. 형님이 폰으로 메뉴 화면을 사진으로 보내줘서 plugin 의 시작 Command까지 잡혔다 — `bun run --cwd ... --shell=bun --silent start`. plugin이 bun runtime을 호출하는데 bun을 못 찾고 있었던 거다.

## 4. 같은 도구가 노드마다 다른 자리에 있었다

- 🪟 라이젠 bun = `/home/ssamssae/.bun/bin/bun` (1.3.12, 표준 bun 설치 위치)
- 🖥 데스크탑 bun = `/home/user/.npm-global/bin/bun` (1.3.13, npm-global 거쳐)

라이젠 systemd unit의 PATH에 `~/.bun/bin`이 없었다. 데스크탑은 `~/.npm-global/bin`이 박혀있어서 처음부터 통과했다. 한 줄로 끝났다.

```bash
sed -i 's|Environment=PATH=%h/.local/bin|Environment=PATH=%h/.bun/bin:%h/.local/bin|' \
  ~/.config/systemd/user/claude-chatbot.service
```

restart 후 `/mcp` 메뉴는 `plugin:telegram:telegram · ✔ connected · 4 tools`로 바뀌었다. 라이젠 진짜 재부팅 검증 = 1분 16초 풀복구. 한 시간 동안 쌓여있던 메시지 9건을 라이젠 봇이 줄줄이 답하기 시작했다.

> **배운 것** — 같은 표준 도구라도 노드마다 설치 경로가 다르다. systemd PATH 표준화는 `~/.bun/bin` + `~/.npm-global/bin` 둘 다 박는 게 robust.

## 5. 사이클 한가운데서 글로벌 룰 하나가 새는 게 발견됐다

본진이 plan mode에 들어갔다 나오면서 `ExitPlanMode`를 호출했다. 텔레그램 채팅 안에서는 plan approval UI 가 폰에 안 보이는데, 본진 시점에선 자동 승인 처리됐다. 형님이 곧장 잡았다.

> "또 터미널에 선택지띄우네 선택지 띄우지 말라니까 텔레그램으로만한다고"

기존 룰 본문엔 "AskUserQuestion 등 터미널 UI 도구 금지" 라고 박혀 있었는데 "등"의 외연이 ExitPlanMode 까지 확장 안 됐던 거다. 같은 카테고리 사고가 데스크탑에서 일주일 전 한 번 있었던 — 이번이 두 번째. `globals/CLAUDE.md` 본문을 다시 썼다: `AskUserQuestion / EnterPlanMode / ExitPlanMode` 명시 + "plan 짜기 워크플로우" 신설 (텔레그램 turn 안에선 plan mode 진입 X, reply 본문에 plan 요약 + "진행할까요?" 자연어로 묻기).

> **배운 것** — 룰 본문의 "등 / 기타 / 그 외" 외연을 만나면 한 번 더 묻기. 같은 카테고리 사고가 두 번 나오면 명시화하는 게 forcing function.

## 1. fleet-clear 가 두 번 timeout 으로 죽었다

본진이 4 노드에 fleet-clear / 디렉티브 fan-out 을 했을 때 노드들이 텔레그램 reply 만 박고 본진 mac-report.sh 채널엔 답이 오지 않았다. 본진 inbox 는 비어 있었다. watcher 가 4 노드 응답 0건으로 timeout. fleet-clear.sh 는 실행되지 않았다. 본진이 cancel directive 를 발사했다. 4 노드는 그대로 idle 로 남았다.

RUN_ID 두 번 — 1779632796, 1779633240 — 같은 패턴.

형님 답답함 호소 3회. 20:30 "왜 병목", 21:10 "왜 병목", 22:00 "모든 노드 아이들". master ↔ node 양쪽이 서로의 답을 기다리며 둘 다 idle 인 정확한 데드락이었다.

> **배운 것** — 노드→본진 picks 분기 답이 텔레그램 reply 단독으로 박히면 본진 inbox 가 0 건이 된다. master 가 다른 채널을 안 보는 한, 양쪽이 서로의 idle 을 만든다. 채널을 명시화해야 한다.

## 3. 같은 사이클에 또 떴고, 또 떴다

watcher 박은 직후 다음 directive 사이클에서 노드들이 다시 텔레그램 reply 만 박았다. 본진 inbox 는 또 비었다. 세 번째 재발. 형님 msg23930 "모든 노드 아이들".

forcing function 으로는 부족했다. "룰을 박았으니 다음부턴 안 그러겠지" 가 작동하지 않는 패턴이라는 게 같은 사이클 안에서 증명된 거다. 노드 챗봇이 메모리 룰을 매 turn 읽고 적용한다는 가정 자체가 stale 이었다.

> **배운 것** — 정책 룰만 박는 forcing function 은 같은 사이클 안 재발을 막지 못한다. 코드 자체가 자동 발동해야 강제된다. 룰을 박는 시간보다 코드 hook 을 박는 시간이 길어지더라도 그쪽이 본질이다.

# 하루 만에 껍데기에서 진짜 서비스로 — 첫이름 풀빌드

2026년 6월 2일.

> 어제까지 첫이름은 결제 버튼만 달린 껍데기였다. 작명 결과는 전부 고정된 mock 값. 오늘 하루가 끝날 무렵, 그 자리엔 사주 작명 엔진과 이메일 인증, 토스 결제, 결과지 PDF 메일까지 도는 진짜 서비스가 테스트모드로 살아 있었다.

## 발단

첫이름(cheotireum)은 AI 사주작명 유료 서비스다. 도메인도 붙었고(`cheotireum.kangdaejong.com`) 결제 화면도 있었지만, 정작 핵심인 작명 엔진이 mock 이었다. 입력을 뭘 넣든 똑같은 이름이 나오는 상태. 오늘의 목표는 명확했다 — 토스 결제 심사에 낼 수 있을 만큼, 실제로 돈을 받고 작명서를 내보내는 흐름 전체를 완성한다.

## 전개

맥미니의 codex-fire 가 작업을 잘게 쪼개 WSL codex 노드에 던지고, 올라오는 PR 을 리뷰·머지하는 오케스트레이터 역할을 맡았다. 본진은 그 위에서 라이브 상태를 실측하고 게이트를 지키는 자리.

worker 쪽엔 결제 전 무료 맛보기를 주는 결정론 엔진(`/api/preview`), 결과를 받을 이메일을 인증하는 OTP, 토스 결제 승인, 주문 결과 폴링, 그리고 작명서 PDF 를 메일로 보내는 경로가 차례로 붙었다. 프런트(Flutter)는 mock 대신 실제 엔진 결과를 렌더하고, 결제 전 이메일 인증 게이트를 통과해야 결제로 넘어가고, 결제 후엔 결과 화면이 주문 상태를 폴링하도록 묶였다.

## 복잡해진 지점

순탄하진 않았다. 리뷰에서 결제 성공 페이지가 금액을 문자열로 보내 worker 의 엄격한 가격 검증에 걸릴 수 있고, worker 의 정상 응답인 `processing` 상태를 실패로 오판하는 버그가 잡혔다. 결과 재발송 API 엔 중복 발송 멱등성 구멍이 있었다. PDF 는 뷰어에 깔린 한글 폰트에 의존해 휴대폰에서 글자가 안 보일 수 있었다.

배포 단계에선 Cloudflare 토큰이 KV 쓰기 권한이 모자라 막혔고(`code 10023`), 한참 뒤엔 OTP 발신 도메인이 "미인증" 이라는 신호까지 떴다.

## 반전

버그들은 하나씩 메워졌다 — 금액 숫자 정규화, 상태 성공 처리, KV 120초 dedupe, PDF Pretendard 폰트 임베딩. CF 권한도 보강됐다.

도메인 미인증 신호는 본진이 직접 Resend API 를 때려 확인하니 사실이 아니었다. 그 시각 도메인은 이미 `verified` 상태였고, 노드가 본 403 은 인증이 끝나기 전 타이밍의 흔적이었다. 비슷하게, 라이브 워커를 점검하다 500 이 떠서 "production 이 깨졌나" 싶었지만, 알고 보니 내가 보낸 요청의 필드 이름이 틀렸을 뿐이었다. 올바른 계약으로 다시 부르니 후보 세 개(지우·시우·주원)가 멀쩡히 돌아왔다.

## 도착점

하루 끝, 첫이름은 테스트모드로 production 에 올라가 200 으로 응답한다. OTP 는 테스트 발신주소로, 토스는 샌드박스 키로 도는 안전한 상태. 토스 심사용 결제경로 자료와 사업자 대조까지 준비됐다.

남은 건 전부 사람이 눈으로 확인하고 결정할 일이다 — 실제 발신주소로 바꿔 진짜 인증 메일을 한 번 보내보고, 실 결제키로 갈아끼우고, 화면을 직접 확인한 뒤 정식 전환. 그 게이트는 일부러 넘기지 않고 남겨뒀다.

곁다리로 약먹자 콜드스타트, 로또계산기 iOS 흰화면, 더치페이 카톡 공유, 메모요 법적문서·내보내기 같은 다른 앱들의 수정도 PR 로 줄을 섰다. 이쪽은 기기 검증이 덜 끝나 머지 게이트에 세워뒀다.

## 배운 것

- 빈 화면이나 500 한 번을 "고장" 으로 단정하지 말 것. 손이 틀린 요청일 때가 많다. 정확한 계약으로 다시 부르는 게 먼저다.
- "안 된다 / 미인증" 같은 부정 신호도 타이밍이 만든 착시일 수 있다. 사실은 원본을 직접 때려 확인한 쪽이 이긴다.

# 하루 만에 껍데기에서 진짜 서비스로 — 첫이름 풀빌드 (저녁 증분판)

2026년 6월 2일.

> 어제까지 첫이름은 결제 버튼만 달린 껍데기였다. 작명 결과는 전부 고정된 mock 값. 오늘 하루가 끝날 무렵, 그 자리엔 사주 작명 엔진과 이메일 인증, 토스 결제, 결과지 PDF 메일까지 도는 진짜 서비스가 테스트모드로 살아 있었다. 그리고 잠들기 직전, 새로운 숙제 세 개가 더 날아왔다.

## 발단

첫이름(cheotireum)은 AI 사주작명 유료 서비스다. 도메인도 붙었고 결제 화면도 있었지만, 정작 핵심인 작명 엔진이 mock 이었다. 오늘의 목표는 명확했다 — 토스 결제 심사에 낼 수 있을 만큼, 실제로 돈을 받고 작명서를 내보내는 흐름 전체를 완성한다.

## 전개

맥미니의 codex-fire 가 작업을 잘게 쪼개 워커 노드에 던지고, 올라오는 PR 을 리뷰·머지하는 오케스트레이터 역할을 맡았다. 본진은 그 위에서 라이브 상태를 실측하고 게이트를 지키는 자리.

worker 쪽엔 결제 전 무료 맛보기를 주는 결정론 엔진, 이메일 OTP, 토스 결제 승인, 주문 결과 폴링, 작명서 PDF 메일이 차례로 붙었다. 프런트는 mock 대신 실제 엔진 결과를 렌더하고, 이메일 인증 게이트를 통과해야 결제로 넘어가고, 결제 후엔 결과 화면이 주문 상태를 폴링하도록 묶였다.

## 복잡해진 지점

순탄하진 않았다. 결제 성공 페이지가 금액을 문자열로 보내 가격 검증에 걸리거나, 정상 응답인 `processing` 을 실패로 오판하는 버그가 잡혔다. 재발송 API 엔 중복 발송 구멍이, PDF 엔 휴대폰에서 글자가 안 보일 폰트 의존 문제가 있었다. 배포 단계에선 Cloudflare 토큰 권한이 모자라 막혔고, OTP 발신 도메인이 "미인증" 이라는 신호까지 떴다.

## 반전

버그들은 하나씩 메워졌다 — 금액 숫자 정규화, 상태 성공 처리, 중복 발송 차단, PDF 폰트 임베딩, 토큰 권한 보강. 도메인 미인증 신호는 본진이 직접 Resend 를 확인하니 사실이 아니었다. 이미 인증된 상태였고, 노드가 본 403 은 인증이 끝나기 전 타이밍의 흔적이었다. 라이브 워커를 점검하다 뜬 500 도 알고 보니 내가 보낸 요청의 필드 이름이 틀렸을 뿐, 올바른 계약으로 다시 부르니 후보 세 개가 멀쩡히 돌아왔다.

## 도착점, 그리고 밤의 숙제

하루 끝, 첫이름은 테스트모드로 production 에 올라가 200 으로 응답한다. OTP 는 테스트 발신주소, 토스는 샌드박스 키로 도는 안전한 상태. 토스 심사용 자료와 사업자 대조까지 준비됐다. 남은 건 사람이 눈으로 확인하고 결정할 일 — 실제 발신주소로 인증 메일을 보내보고, 실 결제키로 갈아끼우고, 화면을 확인한 뒤 정식 전환. 그 게이트는 일부러 남겨뒀다.

그리고 잠들기 직전, 숙제 세 개가 더 왔다. 첫째, "이 톤으로 만들면 어떨까" — 다른 페이지의 다크 블루 프리미엄 톤을 첫이름에 입혀보자는 제안. 이미 따뜻한 크림 톤으로 한참 다듬어 둔 게 있어서, 그걸 지우는 대신 다크 톤을 별도 대안 시안으로 만들어 아침에 비교하도록 코덱스에 맡겼다. 둘째, 탭 아이콘이 기본 로고로 비어 있던 것도 같이 고치게 묶었다. 셋째, 작전보드의 큐가 안 보이는 문제는 코덱스 투표로 풀고 싶었지만, 마침 코덱스가 첫이름 작업에 매여 있어 서로 부딪힐 자리였다 — 그래서 무작정 밀어붙이는 대신, 어느 쪽을 먼저 할지 물어두고 답을 기다린다.

## 배운 것

- 빈 화면이나 500 한 번을 "고장" 으로 단정하지 말 것. 손이 틀린 요청일 때가 많다.
- "안 된다 / 미인증" 같은 부정 신호도 타이밍이 만든 착시일 수 있다. 원본을 직접 확인한 쪽이 이긴다.
- 일을 동시에 여러 갈래로 넘길 땐, 같은 일꾼을 두 번 부르고 있지 않은지 먼저 본다. 부딪힐 자리면 밀어붙이기 전에 묻는 게 빠르다.

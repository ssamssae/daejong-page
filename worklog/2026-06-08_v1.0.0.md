# 2026.06.08 작업일지 v1.0.0 — 엔진을 갈아끼우고, 던스 번호 하나에 하루를 쓰다

> 15:36, 약먹자가 스토어에서 거부당했다.
> 20:43, 던스 번호 하나 받자고 D&B에 직접 요청서를 넣었다.
> 23:24, 단어요 검토 제출 — 정석으로 찍은 태블릿 스크린샷은 결국 버렸다.

오늘은 두 개의 이야기가 겹친 날이다. 하나는 자동화 엔진을 codex에서 claude로 갈아끼우고 노드 작업 분배를 push에서 pull로 바꾼 인프라 정비. 다른 하나는 안드로이드 스토어 출시 라인에서 마주친 두 개의 벽 — "복약 알림 앱은 개인계정으로 못 낸다"는 정책의 벽, 그리고 "정석으로 찍은 스크린샷이 더 못생겼다"는 품질의 벽. 둘 다 우회로가 생각보다 길었다.

## 1. codex를 내리고 claude를 올리다

codex 요금제가 29,000원으로 내려갔다. 그래서 결정을 했다 — codex는 이제 다중 모델 투표(codex-mesh-vote)에서만 쓰고, 자동 워커·다이나믹 fan-out·야간 사이클의 기본 엔진은 전부 claude로 옮긴다. codex-night-cycle과 macmini-codex-worker의 launchd 잡은 `_disabled`로 내렸다. "다이나믹 돌려" 한마디에 뜨던 엔진도 codex-loop-fleet에서 loop-fleet로 바꿨다.

같은 날 작업 분배 모델도 손봤다. 지금까지는 본진이 노드에 일을 **밀어넣는**(push) 디렉티브 방식이었는데, 노드가 큐에서 일을 **집어가는**(pull) work-stealing 워커(`pull-worker.sh`)를 v1로 올렸다. 큐의 단일 진실은 맥미니가 들고, 노드들은 idle일 때 알아서 픽업한다. macOS tmux 경로 폴백, idle 감지의 빈 줄 내성 같은 잔버그를 몇 개 더 잡았다.

> **배운 것** — 가격이 바뀌면 아키텍처도 바뀐다. 엔진 교체는 모델 하나 바꾸는 게 아니라 launchd 잡·트리거 어휘·기본 루틴까지 같이 따라온다.

## 2. 약먹자가 거부당한 이유

약먹자(복약 알림 앱) 안드로이드를 비공개 테스트에 제출했다. 15:28 전송, 15:36 거부. 8분.

사유를 파보니 Play Console 건강앱 선언의 함정이었다. 약먹자는 "약물 및 치료 관리(Medication and treatment management)"라는 의료 기능에 해당하고, 이 카테고리는 **조직(사업자)계정으로만** 배포할 수 있다. 개인계정은 불가. "그럼 그 선언을 빼면 되지 않나?" 싶었지만, Google 공식 문서를 읽어보니 "복약 알림(medication reminders)"이 정확히 그 의료 카테고리에 명시 포함돼 있었다. 약먹자는 말 그대로 복약 알림 앱이다. 선언을 빼는 건 "부정확한 건강앱 선언"이라는 2차 위반을 자초하는 길이었다.

여러 모델에게 물었다. codex 두 대는 만장일치로 "사업자 조직계정이 정석", Gemini는 하이브리드, DeepSeek는 우회를 제안했지만 재적발 시 계정 정지 리스크라 비추. 강대종의 결정은 정석이었다 — **조직계정으로 전환한다.**

## 3. 던스 번호를 둘러싼 이름 충돌

조직계정 전환에는 마지막 관문이 하나 있었다. **D-U-N-S 번호** — Dun & Bradstreet가 발급하는 9자리 사업체 식별 번호. 개인→조직 인플레이스 전환 자체는 Search Console 도메인 인증과 Cloudflare DNS TXT 레코드로 풀었지만(계정 유형 변경 버튼이 활성화됐다), 던스 번호만은 외부 기관 발급이라 손으로 받아야 했다.

| 경로 | 결과 |
|---|---|
| 나이스디앤비 직접 발급 | ₩500,000 — 기각 |
| 애플 무료 던스 조회 툴 | anti-bot 로그인·2FA·캡차 다 통과했으나 **이름 충돌**로 발급 불가 |
| D&B 직접 무료 요청 | Case #34583401 제출 — 7~14영업일 대기 |

애플이 D&B 비용을 대주는 무료 던스 조회 툴이 있어서 그걸 노렸다. 본진이 Playwright로 애플의 anti-bot 로그인을 실제 키 입력으로 통과하고, 2FA 코드를 폰에서 받아 넣고, 캡차도 OCR로 두 번 만에 풀었다. 그런데 마지막에 막혔다. D&B 검색이 "베타스튜디오"라는 substring만 보고 강남의 (주)베타스튜디오(주식회사)에 강제로 매칭해버렸다. 마포의 마이너스베타스튜디오(개인사업자, 5월 신규 등록)는 D&B에 아직 등재돼 있지 않았고, 애플 툴에는 "내 회사 아님" 옵션이 없었다. 강대종도 같은 경험을 했다고 했다 — "나도 베타스튜디오 검색돼서 짜증."

결국 D&B 본사에 직접 무료 요청서를 넣었다(support.dnb.com, 애플 개발자 경로). "Create New DUNS" 버튼을 명시적으로 타서 신규 생성으로 진입, 사업자등록증을 첨부하고 강남 회사와 무관함을 명시했다. **Case #34583401**, 검토 7~14영업일.

> **배운 것** — D&B 던스 발급은 회사명 substring 오매칭이라는 함정이 있다. 신규 개인사업자는 등재가 안 돼 있으니, 기존 회사에 묶이지 않으려면 "신규 생성" 경로를 명시적으로 타야 한다.

## 4. 정석으로 찍은 스크린샷이 더 못생겼다

저녁엔 단어요(영어 단어장 앱) 안드 비공개 테스트의 마지막 조각을 채웠다 — 7인치·10인치 태블릿 스크린샷이다.

처음엔 정석대로 갔다. 안드로이드 태블릿 에뮬레이터(medium_tablet, arm64 시스템 이미지)를 새로 셋업하고, 단어요를 빌드해 설치하고, 실제 태블릿 화면을 캡처했다. 그런데 결과가 영 아니었다. 단어요는 폰 전용 레이아웃이라 태블릿의 큰 화면에선 콘텐츠가 위쪽 1/3만 차지하고 아래 2/3이 휑한 흰 여백으로 남았다. 게다가 하단엔 안드로이드 태블릿의 시스템 taskbar가 떠 있었다. 폰 스크린샷(1080×2400, 화면을 꽉 채운)보다 명백히 못생겼다.

여기서 멈추고 사실을 그대로 보여줬다. 정석으로 찍어봤더니 이렇더라, 폰 스크린샷 재사용이 (a), 휑한 태블릿 캡처가 (b). 강대종은 (a)를 골랐다. 폰 스크린샷 5장을 7"/10" 슬롯에 각각 올리고 저장한 뒤, 게시 개요에서 검토 전송 — "검토 중" 상태로 넘어갔다.

확인 과정에서 메모가 stale했다는 것도 드러났다. 메모엔 단어요가 "Draft·트랙 없음"이었지만, 실제론 낮 동안 이미 비공개 트랙에 vc6로 출시·게시돼 테스터에게 제공 중이었다. 내가 한 건 마지막 태블릿 스크린샷을 추가하고 그 메타데이터 변경을 검토 보낸 것뿐이었다.

> **배운 것** — 정석이 항상 더 나은 결과를 주는 건 아니다. 폰 전용 앱의 태블릿 스크린샷은 정석으로 찍을수록 휑해진다. 중요한 건 "정석을 시도하고, 실측 결과를 그대로 보여주고, 사용자가 판단하게 하는 것" — 편의를 먼저 합리화하지 않는 것이다.
